03-2 — 상세

스트리밍 구현 상세

업로드: aiofiles + 청크

async def save_file_chunked(file, ...):
    async with aiofiles.open(path, "wb") as f:
        while True:
            chunk = await file.read(chunk_size)
            if not chunk:
                break
            await f.write(chunk)
            total += len(chunk)

            # 실시간 크기 체크
            if total > MAX_SIZE:
                file_path.unlink()  # 불완전 파일 삭제
                raise BadRequestException(...)

다운로드: StreamingResponse

async def stream_file(...) -> AsyncGenerator:
    async with aiofiles.open(path, "rb") as f:
        while True:
            chunk = await f.read(chunk_size)
            if not chunk:
                break
            yield chunk  # generator!

return StreamingResponse(
    content=stream_file(...),
    headers={"Content-Length": str(size)}
)

yield로 청크를 하나씩 전달 → 전체 파일을 메모리에 올리지 않음

03-3 — 핵심 구현

프로덕션 강화

Circuit Breaker

3-state 머신으로 장애 전파 차단

CLOSED (정상) ↓ 실패 5회 OPEN (차단) ↓ 30초 후 HALF-OPEN (시험) ↓ 성공 3회 CLOSED

Rate Limiting

IP 기반 슬라이딩 윈도우

# 100 req / 60s per IP
timestamps = defaultdict(deque)

if len(window) >= max_requests:
    return 429 + Retry-After

고정 윈도우의 버스트 문제를 로그 방식으로 해결

운영 안전성

  • Request Timeout
    asyncio.wait_for (30s)
  • Graceful Shutdown
    active_requests 카운터
    + wait_for_drain()
  • Health Check
    /live (프로세스)
    /ready (DB 연결)
04 — DB 최적화

N+1 쿼리 최적화 & 커넥션 풀

N+1 문제 해결

Before: Project 1개 조회 → ProjectModel N개 개별 쿼리

1 + N 쿼리 실행

After: joinedload / selectinload로 한 번에 로딩

1~2 쿼리로 해결

# CRUDBase.get() 에 options 파라미터 추가
crud.get(db, id, options=[
    joinedload(Project.models)
])

상세 조회에서만 적용 — 목록 조회는 불필요한 JOIN 방지

커넥션 풀 설정

파라미터값의미
pool_size20상시 유지 커넥션
max_overflow30최대 추가 커넥션
pool_timeout10s대기 타임아웃
pool_recycle3600s커넥션 재생성 주기

설계 근거

  • pool_recycle: MariaDB의 wait_timeout 대응
  • max_overflow: 트래픽 스파이크 시 탄력적 확장
  • pool_timeout: 데드락 방지
05 — 에러 처리

전역 에러 핸들링

일관된 에러 응답 포맷

{
  "status_code": 404,
  "detail": "User with id '42' not found",
  "error_code": "NOT_FOUND"
}

핸들러 체인 (8개)

예외코드용도
NotFoundException404리소스 미존재
BadRequestException400잘못된 요청
TokenError401인증 실패
ConflictException409데이터 충돌
IntegrityError409DB 무결성
OperationalError503DB 연결
TimeoutError504요청 시간초과
Exception500미처리 예외

설계 원칙

1. 스택 트레이스 미노출

클라이언트에게 내부 구현 세부사항 숨김.
서버 로그에만 기록.

2. 에러 코드 표준화

error_code 필드로 프로그래밍 방식 처리 가능.
클라이언트가 status_code + error_code로 분기.

3. SQLAlchemy 예외 변환

IntegrityError → 409, OperationalError → 503
DB 계층 예외를 HTTP 시맨틱으로 자동 변환.

Q. 왜 HS256을 선택했나?

A. 단일 서버 환경에서 비대칭 키(RS256)는 키 관리 복잡성만 증가. 추후 마이크로서비스 전환 시 RS256으로 변경하면 된다.

  • 면접에서 "왜?"에 즉시 답변 가능
