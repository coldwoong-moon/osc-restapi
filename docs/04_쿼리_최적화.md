# N+1 쿼리 문제 분석 및 최적화

---

## Q: 왜 N+1 쿼리 분석을 진행했는가?

현재 `CRUDBase.get_multi()`는 단순 `select(self.model)` 쿼리를 실행하며,
SQLAlchemy의 기본 lazy loading 전략(`lazy="select"`)을 그대로 사용한다.

API 응답을 직렬화할 때 Pydantic 스키마가 relationship 필드에 접근하면,
SQLAlchemy는 각 레코드에 대해 별도의 SELECT를 추가 실행한다.
데이터가 소량일 때는 체감되지 않지만, 레코드 수가 늘어날수록
DB 왕복 횟수가 선형으로 증가해 응답 시간이 급격히 악화된다.

이 분석은 문제를 **정량적 증거**로 확인하고, 어느 지점에서 최적화가 필요한지
명확히 파악하기 위해 진행했다.

---

## Q: 어떤 목표를 가지고 진행했는가?

1. **N+1 발생 지점 식별** — 현재 코드에서 쿼리가 실제로 몇 회 실행되는지 측정한다.
2. **최적화 전후 비교** — eager loading 적용 전후의 쿼리 수를 숫자로 제시한다.
3. **전략 선택 기준 제시** — `joinedload`와 `selectinload` 중 어느 상황에서 무엇을 써야 하는지 기준을 정리한다.
4. **회귀 방지** — 테스트 코드로 쿼리 수 목표값을 고정해, 이후 변경에서 N+1이 재발하면 즉시 탐지한다.

---

## Q: N+1 쿼리란 무엇이며 현재 코드에서 어디서 발생하는가?

**N+1 쿼리**란 목록 1회 조회(1) 후, 각 레코드의 연관 데이터를 N번 추가 조회하는 패턴이다.
N개의 레코드를 처리할 때 총 N+1회의 DB 왕복이 발생한다.

### 현재 코드의 발생 지점

| 모델 | relationship | 접근 위치 | 쿼리 수 (N=5) |
|------|-------------|-----------|--------------|
| `Project` | `models` (→ `ProjectModel`) | 프로젝트 목록 API 직렬화 시 | 1 + 5 = **6회** |
| `CarryInRequest` | `items` (→ `CarryInRequestItem`) | 반입요청 목록 API 직렬화 시 | 1 + N회 |
| `User` | `partner` (→ `Partner`) | 사용자 목록 API 직렬화 시 | 1 + N회 |
| `StandardCrane` | `crane_items`, `cranes` | 크레인 목록 API 직렬화 시 | 1 + 2N회 |

**근본 원인**: `CRUDBase.get_multi()`가 `select(self.model)`만 실행하고
relationship을 함께 로드하지 않아, Pydantic 직렬화 단계에서 lazy loading이 트리거된다.

```python
# 현재 패턴 (app/crud/base.py)
def get_multi(self, db, *, skip=0, limit=100, filters=None):
    query = select(self.model)          # relationship 없이 단순 SELECT
    results = list(db.scalars(query.offset(skip).limit(limit)).all())
    return results, total              # 반환 후 .models 접근 시 N번 추가 쿼리 발생
```

## Q: 최적화 전후 쿼리 수 비교 결과는?

### Project (5개) + models (각 3개) 기준

| 전략 | 실행 쿼리 수 | 비고 |
|------|------------|------|
| lazy loading (현재) | **6회** | SELECT project 1 + SELECT model×5 |
| `joinedload` | **1회** | LEFT OUTER JOIN으로 한 번에 조회 |
| `selectinload` | **2회** | SELECT project 1 + SELECT model WHERE id IN (...) 1 |

### CarryInRequest (4개) + items (각 3개) 기준

| 전략 | 실행 쿼리 수 |
|------|------------|
| lazy loading (현재) | **5회** |
| `selectinload` | **2회** |

### 규모에 따른 쿼리 수 증가 (lazy loading)

| 레코드 수 | 쿼리 수 |
|----------|---------|
| 10 | 11회 |
| 50 | 51회 |
| 100 | 101회 |
| 1,000 | 1,001회 |

`joinedload`와 `selectinload`는 레코드 수에 무관하게 **1~2회 고정**이다.

## Q: joinedload vs selectinload 어떤 것을 사용해야 하는가?

### joinedload

```python
stmt = select(Project).options(joinedload(Project.models))
```

- SQL: `LEFT OUTER JOIN project_model ON project.id = project_model.project_id`
- **장점**: 쿼리 1회, 네트워크 왕복 최소화
- **단점**: 연관 레코드가 많으면 카르테시안 곱으로 결과셋이 커진다.
  `.distinct().unique()`를 함께 사용해야 중복 제거 가능
- **적합한 상황**: `to-one` 관계 (`user.partner`, `project_model.project`),
  또는 연관 레코드가 적은 `to-many` 관계

### selectinload

```python
stmt = select(CarryInRequest).options(selectinload(CarryInRequest.items))
```

- SQL: `SELECT * FROM carry_in_request_item WHERE carry_in_request_id IN (1, 2, 3, 4)`
- **장점**: 결과셋 중복 없음, 메모리 효율적, 쿼리 2회로 고정
- **단점**: 쿼리가 1회 더 필요
- **적합한 상황**: `to-many` 관계 (`project.models`, `carry_in_request.items`, `standard_crane.crane_items`)

### 권장 선택 기준

```
to-one  관계 → joinedload   (user.partner, crane.standard_crane)
to-many 관계 → selectinload (project.models, carry_in_request.items)
중첩 관계    → selectinload (N+2 수준으로 안전하게 처리)
```

### 적용 예시

```python
# app/crud/project.py
class CRUDProject(CRUDBase[Project, ProjectCreate, ProjectUpdate]):
    def get_multi_with_models(self, db: Session, *, skip=0, limit=100):
        stmt = (
            select(Project)
            .options(selectinload(Project.models))
            .offset(skip)
            .limit(limit)
        )
        return list(db.scalars(stmt).all())
```

## Q: 비즈니스 관점에서의 영향은?

### 응답 시간

현재 프로젝트 목록 100건을 조회할 때 `models` relationship을 직렬화하면
101번의 DB 왕복이 발생한다. MariaDB 기준 왕복당 ~1ms로 가정하면
relationship 접근만으로 **+100ms**가 순수하게 추가된다.

`selectinload` 적용 후에는 이 100ms가 **2ms 이하**로 감소한다.

### 운영 환경에서의 위험

- **커넥션 풀 소진**: N+1은 단일 요청에서 DB 커넥션을 오래 점유한다.
  동시 사용자가 늘면 커넥션 풀이 고갈될 수 있다.
- **슬로우 쿼리 로그**: 100회 이상의 단순 SELECT는 APM에서 감지하기 어려워
  병목 파악이 늦어진다.
- **스케일링 한계**: DB 쿼리 수는 애플리케이션 수평 확장으로 해결되지 않는다.

### 우선순위

| 엔드포인트 | 관계 | 위험도 | 권장 전략 |
|-----------|------|--------|----------|
| `GET /projects` | `models` (1:N) | 높음 | `selectinload` |
| `GET /carry-in-requests` | `items` (1:N) | 높음 | `selectinload` |
| `GET /users` | `partner` (N:1) | 중간 | `joinedload` |
| `GET /standard-cranes` | `crane_items`, `cranes` (각 1:N) | 높음 | `selectinload` 2개 체인 |

### 회귀 방지

`tests/test_query_optimization.py`의 `test_query_count_comparison`가
쿼리 수 목표값(`lazy=6, joined=1, selectin=2`)을 고정한다.
이후 CRUD 변경 시 이 테스트가 실패하면 N+1이 재도입된 것을 즉시 탐지할 수 있다.
