# N+1 쿼리 최적화 적용

---

## Q: 왜 N+1 쿼리 최적화를 적용했는가?

이전 분석(docs/04_쿼리_최적화.md)에서 `CRUDBase.get_multi()`의 lazy loading이
N+1 쿼리를 발생시킨다는 것을 정량적으로 확인했다.
데이터가 소량일 때는 체감되지 않지만, 레코드 수가 늘어날수록 DB 왕복 횟수가
선형으로 증가해 응답 시간이 급격히 악화된다.

프로덕션 배포 전 선제적으로 적용하여 스케일링 병목을 방지하고,
이후 코드 변경에서 N+1이 재발하지 않도록 테스트로 회귀를 방지한다.

---

## Q: 어떤 목표를 가지고 진행했는가?

1. **CRUDBase 확장** — `get()` 메서드에 선택적 `options` 파라미터를 추가하여 eager loading을 지원한다.
2. **하위 호환성 유지** — `options=None`일 때 기존 `db.get()` 경로를 그대로 유지한다.
3. **최소 변경 원칙** — 상세 조회가 필요한 엔드포인트에서만 옵션을 전달하는 구조로,
   기존 코드를 최대한 건드리지 않는다.
4. **회귀 방지** — eager loading 적용 전후의 쿼리 수를 테스트로 고정한다.

---

## Q: 왜 상세 조회에만 적용했는가?

목록 조회(`GET /projects`)는 프로젝트의 기본 정보만 반환하며,
하위 엔티티(모델, 파트 등) 데이터가 응답 스키마에 포함되지 않는다.
이 경우 JOIN을 추가하면 불필요한 데이터를 읽어 오히려 성능이 저하된다.

상세 조회(`GET /projects/{id}`)에서만 하위 엔티티가 필요하며,
이 지점에서만 eager loading을 적용해 목록 조회의 성능을 유지한다.

| 엔드포인트 | 관계 필요 여부 | eager loading 적용 |
|-----------|-------------|------------------|
| `GET /projects` | 불필요 | 미적용 (기존 경로 유지) |
| `GET /projects/{id}` | 필요 (`models`) | `selectinload` 적용 |
| `GET /assemblies/{id}` | 필요 (`parts`) | `selectinload` 적용 |
| `GET /users/{id}` | 필요 (`partner`) | `joinedload` 적용 |

## Q: joinedload vs selectinload 선택 기준은?

분석(docs/04_쿼리_최적화.md)에서 도출한 기준을 그대로 적용했다.

### joinedload — 1:1 / N:1 관계

```python
# User → Partner (N:1)
options = [joinedload(User.partner)]
```

- SQL: `LEFT OUTER JOIN partner ON user.partner_id = partner.id`
- 단일 JOIN 쿼리로 처리, 네트워크 왕복 1회
- 연관 레코드가 1건이므로 카르테시안 곱 문제 없음

### selectinload — 1:N 관계

```python
# Project → Models (1:N)
options = [selectinload(Project.models)]

# Assembly → Parts (1:N)
options = [selectinload(Assembly.parts)]
```

- SQL: `SELECT * FROM project_model WHERE project_id IN (1)`
- 쿼리 2회 고정, 카르테시안 곱 없음
- 1:N에서 `joinedload`를 쓰면 연관 레코드 수만큼 결과셋이 중복되어 메모리 낭비

```
1:1 / N:1 관계 → joinedload   (User→Partner, ProjectModel→Project)
1:N 관계       → selectinload (Project→Models, Assembly→Parts)
```

## Q: CRUDBase 확장 방식을 선택한 이유는?

기존 `db.get(Model, id)`는 SQLAlchemy의 기본 `Session.get()`을 그대로 사용하며
`options()` 체이닝을 지원하지 않는다. eager loading을 적용하려면
`select(Model).where(Model.id == id).options(...)` 형태의 쿼리가 필요하다.

`CRUDBase.get()`에 `options` 파라미터를 추가하여 두 경로를 분기했다.

```python
# app/crud/base.py
def get(
    self,
    db: Session,
    id: int,
    options: list | None = None,
) -> ModelType | None:
    if options is None:
        return db.get(self.model, id)          # 기존 경로 유지 (하위 호환)
    stmt = select(self.model).where(self.model.id == id).options(*options)
    return db.scalars(stmt).unique().first()    # eager loading 경로
```

각 라우터에서 필요한 옵션만 전달한다.

```python
# app/api/routes/projects.py
@router.get("/{project_id}", response_model=ProjectDetail)
def get_project(project_id: int, db: Session = Depends(get_db)):
    project = crud.project.get(
        db,
        id=project_id,
        options=[selectinload(Project.models)],  # 상세 조회에서만 옵션 전달
    )
    if not project:
        raise HTTPException(status_code=404, detail="프로젝트를 찾을 수 없음")
    return project
```

`options=None`일 때 기존 `db.get()` 경로를 유지하므로 다른 라우터는 수정이 불필요하다.
