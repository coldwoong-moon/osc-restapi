# FastAPI 전환 시 고려사항 종합 문서

> 이 문서는 Spring Boot + MyBatis 기반 RPMS 서버를 FastAPI + SQLAlchemy로 전환하면서
> 고려했어야 할 포인트를 Q&A 형식으로 정리한 것입니다.

---

## 1. ORM 전환: MyBatis → SQLAlchemy

### Q: 왜 이 작업을 진행했는가?
A: 기존 RPMS 서버는 MyBatis XML 매퍼 36개 이상을 사용하여 SQL을 직접 관리했다. FastAPI 생태계에서는 SQLAlchemy가 사실상 표준 ORM이며, MyBatis의 XML 기반 SQL 매핑을 Python 생태계로 직접 이식하는 것은 비효율적이다. SQLAlchemy의 선언적 모델 정의와 세션 관리를 통해 타입 안전성과 코드 가독성을 확보하고자 했다.

### Q: 어떤 목표를 가지고 진행했는가?
A: 33개 SQLAlchemy 모델 클래스를 DB 스키마와 1:1로 매핑하되, 기존 osc-restapi의 구조적 문제(각 모델 파일마다 별도 Base 클래스 사용)를 해결하여 단일 `DeclarativeBase`를 공유하는 것이 목표였다.

### Q: 전환 과정에서 어떤 문제가 발생했는가?
A:
- **컬럼명 불일치**: Spring Boot에서 camelCase 컬럼명(`isAccountNonExpired`, `authorityName` 등)을 사용했는데, Python 컨벤션(snake_case)과 충돌한다. DB 호환성을 위해 camelCase를 그대로 유지해야 했다.
- **BLOB 타입 password**: user 테이블의 password 컬럼이 `blob` 타입으로 저장되어 있어, SQLAlchemy에서 `LargeBinary`로 매핑해야 했고 bcrypt 검증 시 bytes 처리가 필요했다.
- **별도 Base 문제**: osc-restapi 원본은 70+개 모델이 각각 별도 Base를 가지고 있어 `metadata.create_all()`이 제대로 동작하지 않는 구조적 결함이 있었다.

### Q: 핵심 교훈은?
A: DB 스키마를 먼저 확인하고 모델을 작성해야 한다. 코드 기반이 아닌 **DB 기준**으로 작업하는 것이 전환 프로젝트에서 가장 안전한 접근법이다. `DESCRIBE table_name`과 실제 데이터 샘플 확인이 선행되어야 한다.

## 2. 인증 체계: Spring Security → 자체 구현

### Q: 왜 이 작업을 진행했는가?
A: Spring Security는 `UserDetailsService`, `AuthenticationManager`, `BCryptPasswordEncoder`, `SecurityContext` 등을 통해 인증/인가를 통합 관리한다. FastAPI에는 이에 대응하는 단일 프레임워크가 없으므로, 인증 로직을 직접 구현해야 했다.

### Q: 어떤 목표를 가지고 진행했는가?
A: 최소한의 인증 엔드포인트(`POST /api/v1/login`)를 제공하되, 원본과 동일한 bcrypt 비밀번호 해싱을 지원하고, 사용자의 authority(권한) 정보를 함께 반환하는 것이 목표였다.

### Q: 전환 과정에서 어떤 문제가 발생했는가?
A:
- **authority 테이블 구조**: `user_authority` 조인 테이블이 존재하지 않았다. 실제로 authority 테이블의 `id` 컬럼이 사용자의 email을 직접 저장하는 구조였다. DB 스키마를 확인하지 않고 일반적인 조인 테이블을 가정했다면 에러가 발생했을 것이다.
- **password BLOB 타입**: `bcrypt.checkpw()`는 bytes를 받는데, SQLAlchemy가 BLOB을 bytes로 반환하므로 별도 인코딩 없이 바로 사용할 수 있었다. 그러나 String으로 잘못 매핑했다면 `encode()` 변환이 필요했다.
- **세션 관리 부재**: 원본은 Redis 기반 세션으로 8시간 유지했지만, 현재 구현은 stateless. 프로덕션에서는 JWT 또는 Redis 세션 추가가 필요하다.

### Q: 핵심 교훈은?
A: 인증 시스템은 프레임워크 의존성이 가장 높은 영역이다. 단순 1:1 포팅이 아니라, 대상 프레임워크의 관용적 방식(FastAPI의 경우 Depends 기반 인젝션, OAuth2PasswordBearer 등)을 따르되, 기존 DB 데이터(bcrypt 해시)와의 호환성을 반드시 검증해야 한다.

## 3. 세션 관리: Redis 기반 → Stateless

### Q: 왜 이 작업을 진행했는가?
A: 원본 시스템은 Redis(127.0.0.1:6379)를 세션 저장소로 사용하며, `HeaderHttpSessionIdResolver.xAuthToken()`으로 X-Auth-Token 헤더 기반 세션을 관리했다. FastAPI는 기본적으로 stateless이므로, 세션 전략을 재설계해야 했다.

### Q: 어떤 목표를 가지고 진행했는가?
A: 1차적으로 stateless API를 구현하고, 향후 JWT 또는 Redis 세션을 추가할 수 있는 구조를 만드는 것이 목표였다. 현재 `/login`은 사용자 정보를 반환만 하고 토큰을 발급하지 않는다.

### Q: 현재 상태의 한계는?
A:
- 인증 상태를 유지할 수 없어, 매 요청마다 인증 정보를 전달해야 함
- 원본의 `session.setMaxInactiveInterval(60*60*8)` 같은 세션 만료 관리가 없음
- 동시 세션 제한(`maximumSessions(1)`)이 구현되어 있지 않음

### Q: 비즈니스 관점에서 왜 중요한가?
A: 인증되지 않은 상태에서 API 접근이 가능하면 보안 위험이 있다. 프로덕션 배포 전 반드시 JWT 또는 세션 기반 인증 미들웨어를 추가해야 한다.

## 4. API 설계: RPC 스타일 → RESTful

### Q: 왜 이 작업을 진행했는가?
A: 원본 RPMS 서버는 `POST /get_users`, `POST /update_user`, `POST /login` 등 RPC 스타일의 URI를 사용했다. 이는 HTTP 메서드의 의미를 무시하고, 모든 요청을 POST로 처리하는 안티패턴이다.

### Q: 어떤 목표를 가지고 진행했는가?
A: RESTful 설계 원칙을 엄격히 준수하는 것이 목표였다:
- **명사형 리소스 URI**: `/users`, `/projects`, `/assemblies`
- **중첩 리소스**: `/projects/{id}/floors`, `/projects/{id}/zones`
- **kebab-case**: `apartment-complexes`, `carry-in-requests`, `part-production-requests`
- **적절한 HTTP 메서드**: GET(조회), POST(생성), PUT(전체 수정), PATCH(부분 수정), DELETE(삭제)
- **의미 있는 상태 코드**: 201+Location(생성), 204(삭제 성공), 404(미발견), 409(충돌)

### Q: 전환 과정에서 어떤 결정이 필요했는가?
A:
- **Upsert 엔드포인트**: 파트 생성/수정을 하나의 `POST` 엔드포인트에서 처리할지, 별도 `PUT`으로 분리할지 결정 필요. GUID 기반 upsert 패턴을 POST에서 처리하기로 결정 (존재하면 업데이트, 없으면 생성)
- **벌크 작업**: 어셈블리 일괄 등록(`POST /projects/{id}/assemblies`)에서 request body를 리스트로 받는 것이 RESTful한지 논쟁. 실용성을 우선하여 리스트 body를 허용
- **PATCH 활용**: 어셈블리의 floor/zone 매핑만 변경하는 경우 PATCH 사용 (`PATCH /assemblies/{guid}/mapping`)

### Q: 핵심 교훈은?
A: REST는 기술이 아니라 아키텍처 스타일이다. URI가 리소스를 표현하고, HTTP 메서드가 행위를 표현하며, 상태 코드가 결과를 표현하는 일관성이 핵심이다. 기존 RPC 스타일을 그대로 옮기는 것은 RESTful이 아니다.

## 5. 스키마 설계: Pydantic v2 활용

### Q: 왜 이 작업을 진행했는가?
A: Spring Boot에서는 Java 도메인 객체가 요청/응답 겸용으로 사용되는 경우가 많았다. FastAPI에서는 Pydantic을 활용하여 요청(Create/Update)과 응답(Response)을 명확히 분리하고, 자동 유효성 검증과 API 문서 생성을 지원한다.

### Q: 어떤 목표를 가지고 진행했는가?
A: 55개 이상의 Pydantic 스키마를 Create/Update/Response로 분리하여:
- 생성 시 필수 필드와 수정 시 선택 필드를 타입 레벨에서 구분
- `ConfigDict(from_attributes=True)`로 ORM 모델 → 응답 자동 직렬화
- DB 컬럼명(camelCase)과 API 필드명(snake_case) 간 alias 매핑

### Q: 전환 과정에서 어떤 문제가 발생했는가?
A:
- **alias와 model_dump() 불일치**: `Field(alias="dept")`로 정의하면 `model_dump()`가 Python 필드명(`department`)을 반환하지만, DB 컬럼은 `dept`이다. `model_dump(by_alias=True)` 또는 `populate_by_name=True` 설정이 필요했다.
- **save_file_name 필드명**: BOM과 Drawing 스키마에서 `saved_file_name`으로 잘못 정의하여 DB 컬럼 `save_file_name`과 불일치 → CRUDBase.create()에서 TypeError 발생. architect 검증에서 발견됨.
- **AssemblyInfoCreate guid alias**: `assembly_guid: str = Field(..., alias="guid")` 사용 시 dump 결과가 `assembly_guid`가 되어 DB 컬럼 `guid`와 불일치.

### Q: 핵심 교훈은?
A: Pydantic alias는 **입력** 방향(JSON → 모델)에서 동작하고, `model_dump()`는 **출력** 방향(모델 → dict)에서 동작한다. 두 방향의 필드명이 일치하는지 반드시 테스트해야 한다. `by_alias` 옵션의 기본값이 False라는 점이 함정이다.

## 6. 에러 처리: 전역 예외 핸들러

### Q: 왜 이 작업을 진행했는가?
A: Spring Boot의 `@ControllerAdvice` + `@ExceptionHandler`에 대응하는 FastAPI의 전역 에러 처리가 필요했다. 일관된 에러 응답 형식(ErrorResponse)으로 클라이언트가 에러를 예측 가능하게 처리할 수 있어야 한다.

### Q: 어떤 목표를 가지고 진행했는가?
A: 3가지 커스텀 예외(NotFoundException, ConflictException, BadRequestException)를 정의하고 각각 404/409/400으로 매핑하여, `{"status_code": int, "detail": str, "error_code": str}` 형태의 일관된 응답을 반환하는 것이 목표였다.

### Q: 현재 상태의 한계는?
A:
- **422 Validation Error**: Pydantic 유효성 검증 실패 시 FastAPI가 자동으로 422를 반환하지만, 형식이 커스텀 ErrorResponse와 다르다. 통일이 필요하다.
- **500 Internal Server Error**: 처리되지 않은 예외에 대한 글로벌 핸들러가 없다. DB 연결 실패, 타임아웃 등에서 스택 트레이스가 노출될 수 있다.
- **IntegrityError 처리**: unique 제약 조건 위반 시 SQLAlchemy IntegrityError가 raw로 전달된다. 이메일 중복 외에도 다른 unique 컬럼에 대한 처리가 필요하다.

### Q: 핵심 교훈은?
A: 에러 처리는 "정상 경로만 동작하면 된다"는 사고에서 벗어나야 한다. 클라이언트(프론트엔드)가 에러를 어떻게 처리할지를 고려하여 일관된 형식으로 설계해야 한다.

## 7. CRUD 패턴: Generic CRUDBase

### Q: 왜 이 작업을 진행했는가?
A: 33개 모델에 대해 각각 CRUD 함수를 작성하면 대량의 보일러플레이트 코드가 발생한다. 제네릭 타입을 활용한 CRUDBase 클래스로 공통 로직을 추출했다.

### Q: 어떤 목표를 가지고 진행했는가?
A: `CRUDBase[ModelType, CreateSchemaType, UpdateSchemaType]` 제네릭 클래스로 get, get_multi, create, update, remove 5가지 기본 연산을 공통화하고, 특수한 로직(upsert, bulk create)만 개별 CRUD에서 오버라이드하는 것이 목표였다.

### Q: 이 패턴의 장단점은?
A:
- **장점**: CRUD 보일러플레이트 90% 이상 제거, 새 모델 추가 시 3줄로 CRUD 완성
- **단점**: 복잡한 쿼리(조인, 서브쿼리, 집계)는 제네릭으로 처리 불가하여 개별 구현 필요. `get_multi()`의 filters 파라미터가 단순 equality 비교만 지원하여 range 쿼리 등에 한계.
- **주의점**: `create()` 메서드에서 `obj_in.model_dump()`를 호출할 때, alias가 아닌 Python 필드명이 반환되므로 DB 컬럼명과 일치해야 한다.

## 8. 비동기 처리: sync vs async

### Q: 왜 이 작업을 진행했는가?
A: FastAPI는 `async def`를 지원하지만, SQLAlchemy의 기본 세션은 동기(sync)이다. 현재 모든 엔드포인트가 `def`(동기)로 정의되어 있어 비동기의 이점을 활용하지 못한다.

### Q: 어떤 목표를 가지고 진행했는가?
A: 1차적으로 동기 구현으로 안정성을 확보하고, 향후 `asyncio` + `AsyncSession` 전환 경로를 문서화하는 것이 목표였다.

### Q: 동기 FastAPI의 동작 방식은?
A: FastAPI는 동기 엔드포인트(`def`)를 스레드 풀에서 실행한다. 기본 스레드 풀 크기는 40이므로, 41번째 동시 요청부터 큐잉이 발생한다. 이는 I/O 바운드 작업(DB 쿼리)에서 병목이 될 수 있다.

### Q: 비동기 전환 시 고려사항은?
A:
- `sqlalchemy.ext.asyncio.AsyncSession` + `create_async_engine` 사용
- `pymysql` 대신 `aiomysql` 드라이버 필요
- 모든 `db.scalar()`, `db.execute()`를 `await`로 변경
- `relationship()` lazy loading이 async에서는 지원되지 않음 → eager loading 필수
- 전환 비용 대비 효과: 동시 사용자 100명 이하에서는 동기로 충분

### Q: 핵심 교훈은?
A: "FastAPI니까 비동기여야 한다"는 오해다. 동시 사용자 수와 I/O 패턴에 따라 동기 구현이 더 단순하고 안정적일 수 있다. 비동기 전환은 **측정된 병목**이 있을 때만 진행해야 한다.

## 9. 마이그레이션 전략: Big Bang vs Strangler Fig

### Q: 왜 이 작업을 진행했는가?
A: 30+ 컨트롤러를 한 번에 전환하는 것(Big Bang)은 위험도가 높다. 점진적 전환(Strangler Fig) 패턴을 검토하여 리스크를 관리할 필요가 있었다.

### Q: 어떤 목표를 가지고 진행했는가?
A: 현재 프로젝트는 학습/재구성 목적이므로 Big Bang 방식을 채택했다. 그러나 프로덕션 환경에서의 전환 전략도 함께 문서화하여 향후 참고자료로 활용하고자 했다.

### Q: 프로덕션에서의 권장 전략은?
A:
1. **1단계**: 읽기 전용 API부터 FastAPI로 전환 (GET 엔드포인트)
2. **2단계**: 리버스 프록시(nginx)에서 URI별 라우팅으로 트래픽 점진 이동
3. **3단계**: 쓰기 API 전환 (POST/PUT/DELETE) - DB 트랜잭션 주의
4. **4단계**: 인증/세션 통합 (가장 마지막)
5. **전 과정**: 양쪽 시스템의 API 응답을 비교하는 Shadow Testing 실행

## 10. 데이터베이스 연결 관리

### Q: 왜 이 작업을 진행했는가?
A: Spring Boot의 HikariCP 커넥션 풀과 SQLAlchemy의 QueuePool은 설정 방식과 기본값이 다르다. 부적절한 풀 설정은 커넥션 고갈이나 리소스 낭비를 초래한다.

### Q: 어떤 목표를 가지고 진행했는가?
A: `pool_pre_ping=True`로 커넥션 유효성 검사를 활성화하고, 적절한 pool_size를 설정하는 것이 목표였다.

### Q: 현재 설정의 문제점은?
A:
- `pool_size` 미지정 → SQLAlchemy 기본값 5. 동시 요청이 5개를 초과하면 대기 발생
- `max_overflow` 미지정 → 기본값 10. 최대 15(5+10) 커넥션까지 확장 가능하지만, 초과 시 `TimeoutError`
- `pool_timeout` 미지정 → 기본값 30초. 커넥션 대기 시 30초 후 에러

### Q: 권장 설정은?
A:
```python
engine = create_engine(
    settings.database_url,
    pool_pre_ping=True,
    pool_size=20,          # 기본 커넥션 수
    max_overflow=30,       # 추가 허용 커넥션
    pool_timeout=10,       # 대기 타임아웃 (초)
    pool_recycle=3600,     # 1시간마다 커넥션 재생성
)
```
