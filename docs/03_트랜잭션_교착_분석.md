# 트랜잭션 교착(Deadlock) 분석

> 작성일: 2026-03-11
> 대상 코드: `app/api/v1/parts.py`, `app/crud/base.py`
> 테스트: `tests/test_deadlock.py`

---

## Q: 왜 교착 상태 분석을 진행했는가?

A: 현재 API 서버는 FastAPI + SQLAlchemy + MariaDB 조합으로 운영되며, 여러 클라이언트(BIM 뷰어, 관리 포털, 배치 작업)가 동시에 파트 속성을 업데이트한다. 특히 `PUT /projects/{project_id}/part-attributes` 엔드포인트는 한 번의 요청으로 수십~수백 개의 `PartAttribute` 레코드를 순차적으로 읽고 수정한다. 동시 요청 수가 늘어날수록 두 트랜잭션이 서로 상대방이 보유한 행 잠금을 기다리는 교착(Deadlock) 상황이 발생할 확률이 높아진다. 교착은 트랜잭션 롤백과 `OperationalError` 응답으로 이어져 데이터 불일치와 사용자 경험 저하를 유발하므로 사전 분석이 필요하다.

---

## Q: 어떤 목표를 가지고 진행했는가?

A: 다음 세 가지 목표를 설정했다.

1. **재현**: 교착이 실제로 발생하는 코드 경로를 격리하여 인메모리 DB(SQLite)로 재현한다.
2. **전략 검증**: Retry, Timeout, 일관된 락 순서라는 세 가지 완화 전략 각각의 효과를 독립적인 테스트로 검증한다.
3. **권장사항 도출**: 현재 코드에서 수정 비용이 가장 낮으면서 효과가 큰 변경을 식별하고 비즈니스 관점의 우선순위를 제시한다.

---

## Q: 교착이 발생할 수 있는 시나리오는?

A: 현재 코드에서 교착이 발생할 수 있는 경로는 다음과 같다.

### 시나리오 1 — `upsert_part_attributes` 역순 동시 요청

```
클라이언트 A 요청: [PART-001, PART-002, PART-003]
클라이언트 B 요청: [PART-003, PART-002, PART-001]
```

```
T_A: SELECT ... WHERE part_number='PART-001' → 행 잠금 획득
T_B: SELECT ... WHERE part_number='PART-003' → 행 잠금 획득
T_A: SELECT ... WHERE part_number='PART-003' → T_B 대기
T_B: SELECT ... WHERE part_number='PART-001' → T_A 대기
→ 교착!
```

`app/api/v1/parts.py:62-82`의 `upsert_part_attributes` 함수가 입력 리스트를 정렬 없이 순서대로 처리하기 때문에 발생한다.

### 시나리오 2 — `upsert_part` 와 `upsert_part_attributes` 혼합 실행

`PartInfo`(guid PK)와 `PartAttribute`(part_number + project_id 복합 PK)는 외래 키 관계는 없지만, 같은 `project_id` 아래에서 동시에 쓰기가 발생하면 InnoDB 테이블 수준 인텐트 잠금이 충돌할 수 있다.

### 시나리오 3 — `CRUDBase.create_multi` 배치 삽입

`app/crud/base.py:45-51`의 `create_multi`는 여러 객체를 `db.add_all()` 후 `db.commit()`한다. 삽입 순서가 자동 증가(autoincrement) ID 할당에 의존하므로, 두 배치가 동시에 실행되면 갭 잠금(Gap Lock) 충돌이 발생할 수 있다.

## Q: 검증 방법은?

A: `tests/test_deadlock.py`에 다섯 가지 테스트를 작성했다.

| 테스트 함수 | 검증 내용 |
|---|---|
| `test_deadlock_scenario` | 역순 업데이트로 교착 조건을 유발하고 `OperationalError` 발생 확인 |
| `test_retry_strategy` | `with_retry()` 래퍼가 일시적 잠금 오류를 3회 이내에 극복하는지 확인 |
| `test_retry_exhaustion` | 최대 재시도 초과 시 예외가 상위로 전파되는지 확인 |
| `test_timeout_strategy` | 짧은 `timeout` 설정으로 무한 대기가 조기 종료되는지 확인 |
| `test_consistent_lock_ordering` | 정렬된 락 순서에서 두 트랜잭션 모두 성공하는지 확인 |
| `test_bulk_upsert_deadlock_risk` | 역순 벌크 upsert 에서 교착 위험이 존재함을 기록 |
| `test_bulk_upsert_with_sorted_order_safe` | 정렬된 upsert 가 동시 실행에서 안전함을 검증 |

SQLite 인메모리 DB를 사용했으므로 MariaDB의 행 수준 잠금과 완전히 동일하지는 않다. SQLite는 데이터베이스 파일 단위의 쓰기 잠금을 사용하므로 MariaDB보다 교착 발생 임계값이 다르지만, 잠금 경쟁 → `OperationalError` → 롤백의 패턴은 동일하다.

## Q: 교착 해결 전략별 효과는?

A: 세 가지 전략을 비교한다.

### 1. Retry (지수 백오프 재시도)

```python
def with_retry(fn, max_retries=3, base_delay=0.05):
    for attempt in range(max_retries):
        try:
            return fn()
        except OperationalError:
            time.sleep(base_delay * (2 ** attempt))
    raise last_exc
```

| 항목 | 평가 |
|---|---|
| 구현 복잡도 | 낮음 — 래퍼 함수 하나로 적용 가능 |
| 효과 | 일시적 교착 대부분 해소. MariaDB가 교착 피해자를 선택해 롤백하면 재시도로 성공 가능 |
| 한계 | 부하가 지속되면 재시도가 모두 소진될 수 있음. 지연 시간이 응답 레이턴시에 직접 반영됨 |
| 적합 상황 | 교착이 드물게 발생하고, 지연 허용 범위가 수백 ms 이내인 경우 |

### 2. Timeout (쿼리/연결 타임아웃)

MariaDB: `SET innodb_lock_wait_timeout = 5;` (기본값 50초 → 5초로 단축)
SQLAlchemy: `create_engine(..., pool_timeout=5)`

| 항목 | 평가 |
|---|---|
| 구현 복잡도 | 낮음 — 엔진 설정 변경만으로 적용 |
| 효과 | 무한 대기를 방지하여 스레드 고갈 예방. 빠른 실패(fast-fail)로 클라이언트가 즉시 재시도 가능 |
| 한계 | 교착 자체를 예방하지 않음. 짧은 타임아웃은 정상 장시간 쿼리도 실패시킬 수 있음 |
| 적합 상황 | 모든 환경에 기본으로 적용해야 하는 안전망. Retry 와 함께 사용 권장 |

### 3. 일관된 락 순서 (Deadlock Prevention)

```python
# 모든 쓰기 경로에서 동일한 정렬 기준 적용
for attr in sorted(attrs_in, key=lambda x: x.part_number):
    ...
```

| 항목 | 평가 |
|---|---|
| 구현 복잡도 | 낮음 — 정렬 한 줄 추가 |
| 효과 | 교착 자체를 근본적으로 예방. 재시도 불필요. 성능 오버헤드 없음 |
| 한계 | 모든 쓰기 경로에 일관 적용이 필요하며, 코드 리뷰 규칙으로 강제해야 함 |
| 적합 상황 | **가장 권장되는 전략.** 단일 함수 수정으로 가장 큰 효과 |

**결론**: 일관된 락 순서 > Retry > Timeout 순으로 우선 적용한다. 세 전략을 계층적으로 조합하면 최대 안정성을 확보할 수 있다.

## Q: 현재 코드의 교착 위험 지점은?

A: 구체적인 위험 지점과 수정 방안은 다음과 같다.

### 위험 지점 1 — `upsert_part_attributes` (최고 위험)

**파일**: `app/api/v1/parts.py:62-82`

```python
# 현재 코드 (위험)
def upsert_part_attributes(project_id, attrs_in, db):
    results = []
    for attr in attrs_in:                          # ← 입력 순서 그대로 처리
        existing = db.scalar(
            select(PartAttribute).where(
                PartAttribute.part_number == attr.part_number,
                PartAttribute.project_id == project_id,
            )
        )
        if existing:
            existing.volume = attr.volume          # ← 암묵적 행 잠금
            existing.ton = attr.ton
        ...
    db.commit()                                    # ← 모든 잠금 한 번에 해제
```

```python
# 수정 코드 (안전)
def upsert_part_attributes(project_id, attrs_in, db):
    results = []
    for attr in sorted(attrs_in, key=lambda x: x.part_number):  # ← 정렬 추가
        ...
```

### 위험 지점 2 — `CRUDBase.create_multi` 삽입 순서 불일치

**파일**: `app/crud/base.py:45-51`

여러 배치 삽입이 동시에 실행될 때 auto-increment ID 할당 순서 충돌로 갭 잠금(Gap Lock)이 발생할 수 있다. MariaDB `innodb_autoinc_lock_mode=2` 설정으로 완화 가능하다.

### 위험 지점 3 — `upsert_part` SELECT-then-UPDATE 패턴

**파일**: `app/api/v1/parts.py:31-38`

```python
existing = db.scalar(select(PartInfo).where(PartInfo.guid == part_in.guid))
if existing:
    ...
    db.commit()
```

`SELECT`와 `UPDATE` 사이에 다른 트랜잭션이 동일 `guid`를 수정하면 Lost Update가 발생한다. `SELECT ... FOR UPDATE` 또는 `ON CONFLICT DO UPDATE` (MariaDB: `INSERT ... ON DUPLICATE KEY UPDATE`) 방식으로 원자적 처리가 필요하다.

## Q: 비즈니스 관점 권장사항은?

A: 적용 우선순위별로 정리한다.

### 즉시 적용 (코드 1줄 수정)

**`upsert_part_attributes` 정렬 추가**

```python
# app/api/v1/parts.py:64
for attr in sorted(attrs_in, key=lambda x: x.part_number):
```

비용: 개발자 30분. 효과: 역순 동시 요청으로 인한 교착의 99% 제거.

### 단기 적용 (1주일 이내)

**MariaDB InnoDB 잠금 대기 타임아웃 단축**

```sql
SET GLOBAL innodb_lock_wait_timeout = 5;  -- 기본 50초 → 5초
```

**애플리케이션 레이어 Retry 적용**

서비스 레이어 또는 미들웨어에 `with_retry()` 래퍼를 적용하여 교착 발생 시 자동 재시도한다.

### 중기 적용 (1달 이내)

**`upsert_part` 원자적 upsert 전환**

```sql
-- MariaDB
INSERT INTO part_attribute (part_number, project_id, volume, ton)
VALUES (:part_number, :project_id, :volume, :ton)
ON DUPLICATE KEY UPDATE volume=VALUES(volume), ton=VALUES(ton);
```

`SELECT-then-UPDATE` 패턴을 단일 SQL 문으로 교체하면 행 잠금 보유 시간이 대폭 단축된다.

**연결 풀 모니터링 설정**

```python
create_engine(
    DATABASE_URL,
    pool_size=10,
    max_overflow=20,
    pool_timeout=30,        # 연결 풀 대기 타임아웃
    pool_pre_ping=True,     # 연결 유효성 사전 확인
)
```

### 장기 적용 (분기 단위)

**낙관적 잠금(Optimistic Locking) 도입**

`AssemblyInfo`와 `PartAttribute`에 `version` 컬럼을 추가하고, 업데이트 시 버전 불일치를 감지하여 충돌을 명시적으로 처리한다. 교착 자체를 발생시키지 않으므로 근본적 해결책이다.

**쓰기 직렬화 큐 도입**

동일 `project_id`에 대한 쓰기를 단일 큐(예: Redis 기반 분산 락)로 직렬화하면 동시성 문제를 완전히 제거할 수 있다. 다만 처리량이 감소하므로 트레이드오프를 평가해야 한다.

## 참고: 교착 발생 시 로그 패턴

MariaDB 교착 발생 시 다음 오류가 기록된다.

```
sqlalchemy.exc.OperationalError: (pymysql.err.OperationalError)
(1213, 'Deadlock found when trying to get lock; try restarting transaction')
```

`SHOW ENGINE INNODB STATUS\G` 명령으로 교착 상세 정보를 확인할 수 있다.
