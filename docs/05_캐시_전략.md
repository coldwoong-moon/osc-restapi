# 캐시 전략 분석

---

## Q: 왜 캐시 전략 분석을 진행했는가?

A: Spring Boot 원본 시스템은 Redis(127.0.0.1:6379, DB 0, 8시간 TTL)를 세션 관리 및 캐시 용도로 사용하고 있었다.
FastAPI로 전환하면서 캐시 레이어가 누락된 채 MariaDB에 직접 모든 요청이 전달되고 있었고,
조회 빈도가 높은 엔드포인트(`/api/v1/projects`, `/api/v1/projects/{id}/assemblies`, `/api/v1/roles`)에서
불필요한 반복 쿼리가 발생하고 있었다.
기능적으로 동일한 데이터를 매 요청마다 DB에서 읽는 패턴이 지속되면 사용자 증가 시 DB 부하가 선형 증가한다.
캐시 전략이 DB 부하를 실질적으로 얼마나 줄일 수 있는지 수치로 확인하고, 구체적인 구현 방향을 결정하기 위해 분석을 진행했다.

---

## Q: 원본 시스템의 Redis 캐시 사용 방식은?

A: Spring Boot 원본 시스템에서 Redis는 두 가지 용도로 사용되었다.

**세션 관리:**
- HTTP 세션 데이터를 Redis에 저장하여 서버 재시작 후에도 세션이 유지되도록 구성
- TTL 8시간 적용 → 업무 시간(8~9시간) 동안 재인증 없이 사용 가능

**데이터 캐시:**
- 역할 목록(`roles`), 프로젝트 목록(`projects`) 등 변경 빈도가 낮고 조회 빈도가 높은 마스터 데이터를 캐시
- 어셈블리 목록, 파트 목록 등 프로젝트별 대량 데이터를 프로젝트 ID 단위로 캐시
- 데이터 변경(등록/수정/삭제) 발생 시 해당 캐시 키를 명시적으로 무효화(이벤트 기반)

이 구조는 단일 Redis 인스턴스로 세션과 캐시를 동시에 처리하는 간결한 설계였다.
FastAPI 전환 버전은 JWT 기반 stateless 인증으로 세션 의존성을 제거했으므로,
Redis 재도입 시 순수 캐시 용도로만 사용하면 된다.

## Q: 현재 FastAPI 시스템에 캐시가 없으면 어떤 문제가 발생하는가?

A: 세 가지 문제가 발생한다.

**1. 동일 쿼리 반복 실행:**
`GET /api/v1/roles`는 역할 목록을 반환하는 엔드포인트로, 데이터가 거의 변경되지 않는다.
캐시 없이 운영하면 100명의 사용자가 동시 접속 시 100번의 동일한 SELECT 쿼리가 DB에 전달된다.

**2. 대량 데이터 반복 직렬화:**
`GET /api/v1/projects/{id}/assemblies`는 프로젝트당 수백~수천 건의 어셈블리를 반환한다.
캐시가 없으면 매 요청마다 DB에서 전체 결과를 읽고 JSON으로 직렬화하는 비용이 반복된다.

**3. DB 부하의 선형 증가:**
사용자 수가 N배 증가하면 DB 쿼리 수도 N배 증가한다.
캐시가 있으면 사용자 수 증가가 DB 부하 증가로 직결되지 않는다(히트된 요청은 DB를 거치지 않음).

시뮬레이션 결과(`test_cache_strategy.py`):
- 파레토 패턴(상위 20% 리소스가 80% 요청 처리) 적용 시 DB 부하 **75~80% 감소** 확인
- 균일 분포에서도 **30~50% 감소** 확인

## Q: 인메모리 캐시 vs Redis 캐시 비교 결과는?

A: 두 방식의 특성이 명확히 다르므로 용도에 따라 선택해야 한다.

| 항목 | 인메모리(TTLCache) | Redis |
|------|-------------------|-------|
| 구현 복잡도 | 낮음 (Python 표준 라이브러리) | 높음 (redis-py, 연결 관리) |
| 응답 속도 | 매우 빠름 (프로세스 내 메모리) | 빠름 (네트워크 왕복 ~1ms) |
| 다중 프로세스/인스턴스 | 불일치 발생 (프로세스마다 별도 캐시) | 일관성 보장 (공유 저장소) |
| 서버 재시작 | 캐시 소멸 | 캐시 유지 |
| 메모리 한계 | 애플리케이션 힙에 종속 | 독립적으로 확장 가능 |
| 외부 의존성 | 없음 | Redis 서버 필요 |
| 운영 비용 | 없음 | Redis 운영 비용 발생 |

**현재 RPMS FastAPI 시스템에 적합한 선택:**

단일 프로세스로 운영하는 초기 단계에서는 인메모리 캐시(`TTLCache` 또는 `cachetools`)가 충분하다.
다중 워커(Gunicorn + Uvicorn) 또는 수평 확장이 필요한 시점에 Redis로 전환한다.
원본 시스템과의 동일 구성을 원하면 Redis를 즉시 도입하는 것이 적합하다.

## Q: 최적 TTL은 얼마인가?

A: 데이터 변경 빈도에 따라 엔드포인트별로 TTL을 다르게 설정해야 한다.

**역할 목록 (`GET /api/v1/roles`):**
- 변경 빈도: 매우 낮음 (시스템 설정 수준)
- 권장 TTL: **1시간 ~ 8시간**
- 근거: 원본 Redis TTL(8시간)과 동일하게 설정 가능. 변경 시 이벤트 무효화 병행.

**프로젝트 목록 (`GET /api/v1/projects`):**
- 변경 빈도: 낮음 (신규 프로젝트 생성은 드문 이벤트)
- 권장 TTL: **30분 ~ 1시간**
- 근거: 프로젝트 생성/삭제 시 이벤트 무효화로 즉시 갱신.

**어셈블리 목록 (`GET /api/v1/projects/{id}/assemblies`):**
- 변경 빈도: 중간 (현장 작업 진행에 따라 상태 변경)
- 권장 TTL: **5분 ~ 15분**
- 근거: 실시간성이 중요한 공정 상태 데이터이므로 짧은 TTL 권장. 어셈블리 수정 API 호출 시 즉시 무효화.

**파트 목록 (`GET /api/v1/projects/{id}/parts`):**
- 변경 빈도: 중간
- 권장 TTL: **5분 ~ 10분**
- 근거: 어셈블리와 동일한 기준 적용.

시뮬레이션(`test_optimal_cache_size.py`)에서 확인한 결과, 캐시 크기가 Hot 리소스 수(20개)를 수용하는 시점부터 히트율이 70% 이상으로 안정화되었고, 그 이상의 크기 증가는 히트율을 5% 미만으로 추가 개선했다(한계 효용 감소).

## Q: 캐시 무효화 전략은?

A: TTL 기반과 이벤트 기반을 계층적으로 결합하는 방식을 권장한다.

**계층 1 - TTL 기반 (안전망):**
모든 캐시 항목에 TTL을 설정하여 이벤트 무효화가 누락되더라도 일정 시간 후 자동 갱신되도록 한다.
TTL은 데이터 신선도 요구사항에 따라 엔드포인트별로 위 기준을 적용한다.

**계층 2 - 이벤트 기반 (즉시 반영):**
데이터 변경 API(`POST`, `PUT`, `PATCH`, `DELETE`)가 호출될 때 관련 캐시 키를 즉시 무효화한다.

```
# 예시: 어셈블리 상태 변경 시
PATCH /api/v1/assemblies/{guid}
  → 처리 완료 후 캐시 무효화:
     - 키: "assemblies:project_{project_id}"
     - 방식: invalidate_pattern("assemblies:project_{project_id}:")
```

**캐시 키 설계 원칙:**
- 엔드포인트 경로 + 쿼리 파라미터를 조합하여 키 생성
- 프리픽스로 리소스 타입을 명시하여 패턴 무효화 가능하게 구성

```
"projects:list"                          # 프로젝트 목록
"assemblies:project_{id}"                # 특정 프로젝트의 어셈블리 목록
"assemblies:project_{id}:page_{n}"       # 페이지네이션 포함
"roles:list"                             # 역할 목록
```

**시뮬레이션 검증 결과:**
`test_ttl_vs_event_invalidation` 테스트에서 이벤트 기반 무효화가 TTL 만료 전 즉시 최신 데이터를 반환함을 확인했다.
TTL만 사용하는 경우 변경 후 최대 TTL 시간 동안 구 데이터(stale)가 반환될 수 있다.

## Q: 비즈니스 관점 권장사항은?

A: 세 단계로 나누어 점진적으로 도입하는 방식을 권장한다.

**단계 1 - 즉시 적용 (인메모리 캐시, 외부 의존 없음):**

`cachetools` 라이브러리의 `TTLCache`를 FastAPI의 `lifespan` 이벤트로 초기화하여 글로벌 캐시로 운용한다.
역할 목록과 프로젝트 목록처럼 변경이 드문 마스터 데이터에 우선 적용한다.
예상 효과: DB 쿼리 50~70% 감소, 응답 속도 30~50% 개선.

```python
# 예시 구현 방향
from cachetools import TTLCache

# app 시작 시 초기화
roles_cache = TTLCache(maxsize=10, ttl=3600)       # 1시간
projects_cache = TTLCache(maxsize=50, ttl=1800)    # 30분
assemblies_cache = TTLCache(maxsize=200, ttl=300)  # 5분
```

**단계 2 - 운영 안정화 (이벤트 무효화 추가):**

변경 API에 캐시 무효화 로직을 추가하여 데이터 신선도를 보장한다.
FastAPI Dependency 또는 미들웨어 패턴으로 구현하면 비즈니스 로직과 캐시 로직을 분리할 수 있다.

**단계 3 - 수평 확장 시 (Redis 도입):**

다중 Uvicorn 워커 또는 다중 서버 인스턴스로 운영할 경우 인메모리 캐시는 프로세스마다 독립적으로 동작하여 데이터 불일치가 발생한다.
이 시점에 Redis(`redis-py` + `aioredis`)로 전환하여 공유 캐시를 구성한다.
원본 Spring Boot 시스템의 Redis 설정(127.0.0.1:6379, DB 0, TTL 8시간)을 참조하되, TTL은 엔드포인트별 권장값으로 세분화한다.

**우선순위 요약:**

| 우선순위 | 대상 | 기대 효과 |
|---------|------|----------|
| P0 (즉시) | `GET /api/v1/roles` | DB 쿼리 거의 제거 |
| P0 (즉시) | `GET /api/v1/projects` | DB 쿼리 70~80% 감소 |
| P1 (1주 내) | `GET /api/v1/projects/{id}/assemblies` | DB 부하 50~70% 감소 |
| P1 (1주 내) | `GET /api/v1/projects/{id}/parts` | DB 부하 50~70% 감소 |
| P2 (수평 확장 시) | 전체 캐시 → Redis 전환 | 다중 인스턴스 일관성 보장 |
