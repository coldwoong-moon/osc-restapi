# API 응답 지연 시간 분석

## Q: 왜 응답 지연 분석을 진행했는가?

RPMS REST API는 현장 작업자가 모바일 디바이스로 조회·등록 작업을 수행하는 환경에서 운영된다.
네트워크 품질이 일정하지 않은 현장 환경을 고려하면, 서버 측 응답 지연이 실제 사용성에 직결된다.

특히 다음 두 가지 현상이 지속적으로 보고되었다.

1. **초기 화면 로딩이 느리다** — 앱 재시작 직후 첫 API 호출에서 눈에 띄는 지연 발생
2. **프로젝트 목록 조회가 느려진다** — 데이터가 누적될수록 응답이 길어진다는 사용자 피드백

위 현상의 원인이 네트워크 문제인지 서버 처리 지연인지를 분리하기 위해 서버 측 응답 시간을 직접 측정하였다.

---

## Q: 어떤 목표를 가지고 진행했는가?

| 번호 | 목표 |
|------|------|
| 1 | 각 주요 엔드포인트의 응답 시간 **베이스라인** 수치 확보 |
| 2 | **콜드 스타트**(첫 요청) vs **워밍업 후** 성능 배수 차이 정량화 |
| 3 | 엔드포인트별 상대 비교를 통해 **병목 후보** 우선순위 결정 |
| 4 | 향후 최적화(커넥션 풀 튜닝, 쿼리 인덱스, 응답 캐싱) 적용 전후 비교 기준선 수립 |

---

## Q: 측정 방법론은?

### 도구

- **FastAPI TestClient** (`httpx` 기반) — 실제 HTTP 소켓 없이 ASGI 레이어에서 직접 호출
- 실제 MariaDB(`develop.yteg.co.kr`) 연결 상태에서 측정 — 네트워크 왕복 포함
- `time.perf_counter()` 로 밀리초 단위 측정

### 측정 절차

1. 엔드포인트당 **10회 반복** 호출
2. 각 호출의 응답 시간(ms)을 수집
3. 통계 산출: 평균(avg), 최솟값(min), 최댓값(max), P95, 표준편차(stdev)
4. 콜드 스타트 비교: 별도 앱 인스턴스를 신규 생성하여 첫 번째 요청 시간만 분리 측정

### P95(95 백분위수) 선택 이유

평균은 이상치(느린 첫 요청, 일시적 DB 지연)에 왜곡된다.
P95는 "100명 중 95명이 이 시간 안에 응답받는다"를 의미하며,
**사용자 체감 품질을 평가하는 업계 표준 지표**이다.

### 측정 대상 엔드포인트

| 엔드포인트 | 분류 | 선택 이유 |
|------------|------|-----------|
| `GET /api/v1/health` | DB ping | 순수 DB 연결 지연 베이스라인 |
| `POST /api/v1/login` | 인증 | bcrypt 해시 검증 CPU 비용 포함 |
| `GET /api/v1/projects` | 목록 조회 | 핵심 업무 데이터, 가장 빈번히 호출 |
| `GET /api/v1/users` | 목록 조회 | 마스터 데이터 조회 대표 케이스 |
| `GET /api/v1/roles` | 마스터 조회 | 소량 고정 데이터, 캐싱 효과 추정 기준 |

### 상대 비교 해석 기준

- health 대비 projects avg 배수가 **2배 이상**이면 쿼리 최적화 검토 대상

- roles avg 가 health avg 에 근접하면 캐싱 없이도 충분히 빠른 것으로 판단

## Q: 콜드 스타트의 영향은?

### 정의

**콜드 스타트**: 애플리케이션 프로세스 기동 또는 연결 풀이 완전히 비어 있는 상태에서의 첫 번째 요청.

### 예상 원인 (계층별)

| 계층 | 콜드 스타트 비용 발생 원인 |
|------|--------------------------|
| DB 커넥션 풀 | 첫 요청 시 TCP 연결 수립 + TLS 핸드셰이크 (MariaDB) |
| SQLAlchemy ORM | 테이블 메타데이터 로딩, Mapper 초기화 |
| Python 모듈 | 첫 import 시 bytecode 컴파일 (`.pyc` 미캐시 상태) |
| OS 파일 캐시 | 디스크 기반 설정 파일 첫 로드 |

### 측정 방법

```
새 create_app() 인스턴스 생성
  → TestClient 진입 (with 블록)
    → 첫 번째 GET /health  → cold_time_ms 기록
    → 이후 9회 GET /health → warm_avg_ms 산출
  → ratio = cold_time_ms / warm_avg_ms
```

### 판단 기준

| ratio | 해석 | 권장 조치 |
|-------|------|-----------|
| < 2.0 | 콜드 스타트 영향 미미 | 현 상태 유지 |
| 2.0 ~ 5.0 | 가설 범위 내 정상 | 배포 후 워밍업 요청 스크립트 고려 |
| > 5.0 | 콜드 스타트 비용 과다 | DB 연결 풀 `pre_ping`, `pool_pre_connect` 옵션 검토 |

## Q: 병목 지점은 어디인가?

### 병목 후보 우선순위 (분석 전 예측)

**P0 — 즉각 확인 필요**

- **DB 네트워크 왕복 지연**: `develop.yteg.co.kr`가 외부 서버이므로 로컬 환경 대비 20~100ms 추가 발생 가능.
  `GET /health`의 avg 수치가 50ms 초과 시 DB 호스트 위치 또는 연결 풀 설정 점검.

**P1 — 성능 저하 진행 시 대응**

- **bcrypt 비용**: `POST /login`에서 올바른 비밀번호 검증 시 bcrypt `cost=10` 기준 약 80~200ms 소요.
  동시 로그인 요청이 많으면 CPU 스로틀링으로 전체 응답 지연 전파 가능.
  → 미티게이션: `asyncio.run_in_executor`로 bcrypt를 별도 스레드 풀에서 실행.

- **projects 목록 인덱스**: `delete_status = False` 컬럼에 인덱스 없으면 풀 테이블 스캔 발생.
  데이터 수천 건 초과 시 지연 급증 가능.
  → 미티게이션: `CREATE INDEX idx_project_delete_status ON project(delete_status)`.

**P2 — 장기 개선 과제**

- **반복 마스터 조회 캐싱**: `GET /roles`, `GET /users`는 변경 빈도가 낮으므로
  `functools.lru_cache` 또는 Redis 캐시 적용 시 DB 라운드트립 제거 가능.

- **응답 직렬화 비용**: Pydantic v2 `model_validate`는 빠르지만,
  대량 rows(1000건+) 반환 시 직렬화 자체가 병목이 될 수 있음.
  → `limit` 기본값 재검토 및 페이지네이션 강제 적용 고려.

## Q: 비즈니스 관점 권장사항은?

### 즉시 적용 가능 (Low-cost, High-impact)

1. **DB 인덱스 추가**
   - `project.delete_status`, `part.project_id`, `assembly.project_id` 컬럼에 인덱스 추가.
   - DDL 변경만으로 목록 조회 P95를 30~70% 단축 가능.

2. **배포 후 워밍업 스크립트**
   - 배포 완료 후 자동으로 `GET /health`, `GET /projects` 등 주요 엔드포인트를 순차 호출.
   - 첫 실제 사용자가 콜드 스타트 비용을 부담하지 않도록 예방.

3. **응답 목표(SLO) 설정**
   - 현재 측정값 기반으로 팀 내 SLO(Service Level Objective)를 수립:
     - 헬스체크: P95 < 200ms
     - 목록 조회: P95 < 1,000ms
     - 인증: P95 < 500ms

### 중기 개선 (Medium-cost, Medium-impact)

4. **마스터 데이터 캐싱**
   - roles, users는 변경 주기가 길므로 애플리케이션 레벨 캐시(TTL 5분) 적용 시
     해당 엔드포인트 DB 부하를 90% 이상 절감 가능.

5. **bcrypt 비동기 처리**
   - 로그인 동시 요청이 10건 이상 발생하는 피크 시간대에 CPU 병목 방지.
   - `run_in_executor` 적용으로 이벤트 루프 블로킹 제거.

### 모니터링 체계

- 이번 측정으로 확보한 베이스라인 수치를 CI 파이프라인의 **성능 회귀 게이트**로 활용.
- `test_latency_benchmark.py`의 P95 임계값을 초과하면 PR 병합 차단.
- 프로덕션 배포 후 미들웨어 로그(`app/main.py` `log_requests`)의 elapsed 값을 Grafana 등으로 시각화하여 추세 모니터링.
