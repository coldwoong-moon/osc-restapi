# 동시성 및 커넥션 풀 분석

---

## Q: 왜 이 분석을 진행했는가?

A: RPMS REST API 운영 중 특정 시간대에 응답 지연과 DB 연결 오류가 산발적으로 발생했다.
로그에서 반복적으로 확인된 패턴은 `QueuePool limit of size N overflow N reached` 에러였고,
이는 SQLAlchemy 커넥션 풀이 고갈될 때 발생하는 시그널이다.
근본 원인이 풀 설정의 문제인지, 아니면 쿼리 자체의 느린 응답인지 명확히 구분하지 못한 채
운영하고 있었기 때문에 정량적 분석이 필요했다.

---

## Q: 어떤 목표를 가지고 진행했는가?

A: 세 가지 구체적인 목표를 설정했다.

1. **현상 재현**: 동시 사용자 증가 시 커넥션 풀 고갈이 실제로 응답 지연과 에러를 유발하는지 확인한다.
2. **설정별 수치 비교**: `pool_size=5 / 20 / 50` 환경에서 동시 요청 처리 성공률과 평균 응답 시간을 측정하여 설정값과 성능 사이의 상관관계를 수치화한다.
3. **권장 설정 도출**: 측정 결과를 바탕으로 RPMS API의 예상 동시 사용자 수에 적합한 `pool_size` / `max_overflow` 권장값을 제시한다.

---

## Q: 가설은 무엇이었는가?

A: 동시 사용자 수가 커넥션 풀 크기(`pool_size + max_overflow`)를 초과하면 두 가지 현상이 나타날 것이라고 가정했다.

- **응답 지연 급증**: 풀이 가득 찬 상태에서 추가 요청은 빈 커넥션이 반환될 때까지 대기(blocking)하므로, 대기 시간이 쌓여 응답 지연이 기하급수적으로 늘어난다.
- **QueuePool overflow 에러 발생**: 대기 시간이 `pool_timeout`(기본 30초)을 초과하면 SQLAlchemy는 `OperationalError`를 발생시킨다. 이때 에러 메시지에는 `QueuePool limit of size N overflow N reached` 문자열이 포함된다.

반대로, `pool_size`를 충분히 크게 설정하면 동일한 부하에서도 에러 없이 모든 요청이 정상 처리될 것이라고 예측했다.

## Q: 테스트 방법론은?

A: 실제 DB(`develop.yteg.co.kr:3306/rpms_250905`)에 의존하지 않고 CI 환경에서 재현 가능하도록
**Mock 기반 시뮬레이션** 방식을 선택했다.

| 항목 | 상세 |
|------|------|
| 테스트 프레임워크 | `pytest` |
| 동시성 구현 | `concurrent.futures.ThreadPoolExecutor` |
| DB 의존성 제거 | `threading.BoundedSemaphore`로 QueuePool 동작을 모의 구현 |
| 커넥션 지연 시뮬레이션 | `checkout_delay=0.02s` (실제 네트워크 레이턴시 근사) |
| 쿼리 실행 시간 시뮬레이션 | `work_duration=0.05~0.3s` |
| overflow 조건 | `semaphore.acquire(timeout=0.5)` 실패 시 `OperationalError` 발생 |

테스트 케이스 구성:

```
test_concurrent_10_sessions   — 소형 풀(5)에서 10 동시 세션 기본 부하
test_concurrent_50_sessions   — 소형 풀(5)에서 50 동시 세션 overflow 검증
test_concurrent_100_sessions  — 대형 풀(50)에서 100 동시 세션 수용 한계 검증
test_pool_size_comparison     — pool_size 5/20/50 비교 (30 동시 요청)
test_pool_overflow_behavior   — overflow 에러 메시지 및 성공 요청 수 검증
```

## Q: 테스트 결과는?

A: 시뮬레이션 결과를 정리하면 다음과 같다.

### pool_size별 성능 비교 (30 동시 요청 기준)

| pool_size | max_overflow | 총 용량 | 성공 수 | 성공률(%) | 평균 응답(s) | 에러 수 |
|:---------:|:------------:|:-------:|:-------:|:---------:|:------------:|:-------:|
| 5         | 5            | 10      | 10      | 33.3      | 0.07         | 20      |
| 20        | 10           | 30      | 30      | 100.0     | 0.09         | 0       |
| 50        | 10           | 60      | 30      | 100.0     | 0.07         | 0       |

### 동시 세션 수별 결과 (pool_size=5 / max_overflow=5 고정)

| 동시 세션 수 | 총 용량 | 성공 수 | 에러 수 | 평균 응답(s) |
|:------------:|:-------:|:-------:|:-------:|:------------:|
| 10           | 10      | 10      | 0       | 0.07         |
| 50           | 10      | ~10     | ~40     | 0.07         |
| 100          | 10      | ~10     | ~90     | 0.07         |

### 주요 관찰 결과

- `pool_size=5, max_overflow=5` 환경에서 30 동시 요청을 받으면 **67%가 QueuePool overflow 에러로 실패**한다.
- `pool_size=20, max_overflow=10`으로 설정하면 30 동시 요청 전부를 에러 없이 처리할 수 있다.
- overflow 에러 발생 시 메시지에는 항상 `QueuePool limit of size N overflow N reached` 가 포함되며, 이를 통해 풀 고갈 여부를 명확히 구분할 수 있다.
- 성공한 요청의 평균 응답 시간 자체는 pool_size에 관계없이 유사하다. 즉, **문제는 응답 속도가 아니라 연결 획득 실패(에러율)** 에 있다.

## Q: 비즈니스 관점에서의 영향은?

A: RPMS는 현장 작업자들이 실시간으로 사용하는 시스템이다.
커넥션 풀 고갈로 인한 에러는 다음과 같은 직접적인 업무 영향을 유발한다.

| 영향 구분 | 상세 내용 |
|-----------|-----------|
| 작업 중단 | 파트 조회 / 어셈블리 상태 확인 실패로 현장 작업이 멈춤 |
| 데이터 불일치 위험 | 트랜잭션이 완료되지 않은 상태로 중단될 경우 DB 정합성 문제 가능성 |
| 오류 누적 | 재시도 로직 없이 에러가 클라이언트에 노출되면 사용자가 중복 조작을 시도하여 부하가 더 증가하는 악순환 발생 |
| 모니터링 사각지대 | `QueuePool limit` 에러가 APM/로그에 집계되지 않으면 문제 발생 시점과 규모를 파악하기 어려움 |

동시 사용자가 10명 수준이라면 `pool_size=5` 설정도 간신히 버틸 수 있지만,
현장 교대 시간(08:00 / 17:00)에 30~50명이 동시에 접속하는 피크 패턴이 확인된다면
현재 기본 설정은 즉각 변경이 필요하다.

## Q: 권장 설정은?

A: 예상 동시 사용자 수에 따른 3단계 권장 설정을 제시한다.

### 설정 권장표

| 운영 규모 | 예상 동시 사용자 | pool_size | max_overflow | pool_timeout | pool_recycle |
|:---------:|:----------------:|:---------:|:------------:|:------------:|:------------:|
| 소규모    | ~20명            | 10        | 10           | 30s          | 1800s        |
| 중규모    | ~50명            | 20        | 20           | 30s          | 1800s        |
| 대규모    | ~100명           | 50        | 30           | 30s          | 1800s        |

### app/db/session.py 적용 예시 (중규모 기준)

```python
from sqlalchemy import create_engine
from app.core.config import settings

engine = create_engine(
    settings.database_url,
    pool_pre_ping=True,   # 유휴 커넥션 자동 헬스체크
    pool_size=20,         # 기본 커넥션 수
    max_overflow=20,      # 피크 시 최대 추가 커넥션
    pool_timeout=30,      # 커넥션 획득 대기 최대 시간(초)
    pool_recycle=1800,    # 30분마다 커넥션 재생성 (MariaDB idle timeout 대응)
)
```

### 추가 권장 사항

1. **`pool_pre_ping=True` 유지**: MariaDB의 `wait_timeout`(기본 8시간) 초과로 끊긴 커넥션을 자동으로 교체한다.
2. **`pool_recycle=1800` 설정**: `wait_timeout` 이전에 주기적으로 커넥션을 갱신하여 `MySQL server has gone away` 에러를 예방한다.
3. **에러 모니터링 추가**: `QueuePool limit` 에러를 별도 카운터로 집계하여 알림을 설정한다.
4. **슬로우 쿼리 병행 점검**: 커넥션 풀을 늘리더라도 슬로우 쿼리가 커넥션을 오래 점유하면 동일 문제가 재발한다. `SHOW PROCESSLIST` 또는 슬로우 쿼리 로그로 병행 점검을 권장한다.
