# 프로덕션 강화 종합

> 이 문서는 실서비스 운영 리스크에 대응하기 위해 진행한
> 6가지 프로덕션 강화 작업의 설계 결정과 검증 결과를 종합한 것입니다.

---

## Q: 왜 프로덕션 강화 작업을 진행했는가?

A: 개발 환경에서는 드러나지 않지만 실서비스에서 반드시 발생하는 세 가지 운영 리스크가 있다.

1. **트래픽 급증**: 동시 사용자가 늘어나면 DB 커넥션 풀이 소진되고,
   처리되지 않은 예외가 스택 트레이스를 그대로 노출하며, 느린 요청 하나가
   전체 스레드 풀을 점유하여 서비스 전체가 멈춘다.

2. **DB 장애**: 외부 DB 서버가 일시적으로 응답하지 않을 때,
   재시도 로직 없이는 모든 요청이 즉시 500 오류로 실패한다.
   반면 적절한 Circuit Breaker 없이는 장애가 전파되어 다운스트림 전체가 무너진다.

3. **배포 시 서비스 중단**: 새 버전 배포 시 프로세스를 즉시 종료하면
   처리 중이던 요청이 강제로 끊긴다. 사용자는 불완전한 응답을 받고,
   DB 트랜잭션은 롤백되지 않은 채 남을 수 있다.

---

## Q: 어떤 목표를 가지고 진행했는가?

A: 총 6가지 개선 항목을 구현하여 운영 중단 없는 안정적인 서비스를 목표로 했다.

| # | 개선 항목 | 해결하는 문제 | 핵심 응답 코드 |
|---|-----------|--------------|---------------|
| 1 | DB 커넥션 풀 최적화 | 동시 접속 시 커넥션 고갈 | `500` 방지 |
| 2 | 전역 에러 핸들러 | 미처리 예외 스택 트레이스 노출 | `500` → 정형화 |
| 3 | Rate Limiting | API 남용, DDoS 완화 | `429` |
| 4 | Request Timeout | 느린 요청의 스레드 점유 | `504` |
| 5 | Circuit Breaker | 장애 전파 방지 | `503` |
| 6 | Graceful Shutdown + Health Check | 배포 시 진행 중 요청 보호 | `503` (셧다운 중) |

## 1. DB 커넥션 풀 최적화

### Q: 왜 커넥션 풀 설정이 필요한가?

A: SQLAlchemy 기본값은 `pool_size=5, max_overflow=10`으로 최대 15개의 동시 커넥션만 허용한다.
동시 사용자가 16명 이상이 되는 순간부터 `pool_timeout` 초 후 `TimeoutError`가 발생하여
서비스가 중단된다. 설정 변경만으로 이 임계값을 수십 배 올릴 수 있다.

### Q: 각 설정값의 근거는?

```python
engine = create_engine(
    settings.database_url,
    pool_pre_ping=True,     # 요청 전 커넥션 유효성 확인 (좀비 커넥션 제거)
    pool_size=20,           # 항상 유지할 커넥션 수
    max_overflow=30,        # 피크 시 추가 허용 커넥션 (최대 50개)
    pool_timeout=10,        # 커넥션 대기 최대 시간 (초)
    pool_recycle=3600,      # 1시간마다 커넥션 재생성 (DB 타임아웃 방지)
)
```

| 파라미터 | 값 | 설정 근거 |
|---------|-----|----------|
| `pool_size` | 20 | 일반적인 웹 서비스의 평균 동시 처리 요청 수 기준 |
| `max_overflow` | 30 | 피크 트래픽 시 2.5배 버스트 허용 (총 50개) |
| `pool_timeout` | 10 | 30초 대기는 사용자 이탈 유발. 10초 후 빠른 실패가 낫다 |
| `pool_recycle` | 3600 | MariaDB 기본 `wait_timeout=28800`보다 짧게 설정하여 끊긴 커넥션 재사용 방지 |
| `pool_pre_ping` | True | `SELECT 1` 핑으로 유효하지 않은 커넥션을 요청 전에 교체 |

### Q: 개선 효과는?

A: 설정 변경 전후 비교:

| 항목 | 변경 전 (기본값) | 변경 후 |
|------|----------------|---------|
| 최대 동시 커넥션 | 15개 | 50개 |
| 동시 사용자 한계 | ~15명 | ~50명 |
| 구현 난이도 | — | **낮음** (설정 1시간) |
| 커넥션 좀비 방지 | 없음 | `pool_pre_ping` |

## 2. 전역 에러 핸들러

### Q: 왜 전역 에러 핸들러가 필요한가?

A: FastAPI의 기본 동작은 처리되지 않은 예외를 `500 Internal Server Error`로 반환한다.
문제는 SQLAlchemy 스택 트레이스, DB 연결 문자열, 내부 경로 등이 응답에 포함될 수 있다는 것이다.
이는 보안 취약점(정보 노출)이자 사용자 경험 저하(기술적 오류 메시지 노출) 요소이다.

### Q: 어떤 예외 유형을 처리하는가?

```python
# 1. DB 무결성 제약 위반 (중복 키, NOT NULL 등)
@app.exception_handler(IntegrityError)
async def integrity_error_handler(request, exc):
    return JSONResponse(status_code=409,
        content={"status_code": 409, "detail": "데이터 무결성 오류가 발생했습니다."})

# 2. DB 연결 오류, 커넥션 풀 소진
@app.exception_handler(OperationalError)
async def operational_error_handler(request, exc):
    return JSONResponse(status_code=503,
        content={"status_code": 503, "detail": "데이터베이스 연결 오류가 발생했습니다."})

# 3. 요청 처리 타임아웃
@app.exception_handler(asyncio.TimeoutError)
async def timeout_error_handler(request, exc):
    return JSONResponse(status_code=504,
        content={"status_code": 504, "detail": "요청 처리 시간이 초과되었습니다."})

# 4. 그 외 모든 미처리 예외
@app.exception_handler(Exception)
async def unhandled_exception_handler(request, exc):
    logger.error("Unhandled exception: %s", exc, exc_info=True)  # 내부 로깅만
    return JSONResponse(status_code=500,
        content={"status_code": 500, "detail": "내부 서버 오류가 발생했습니다."})
```

| 예외 유형 | HTTP 상태 코드 | 클라이언트 메시지 | 내부 처리 |
|-----------|--------------|-----------------|----------|
| `IntegrityError` | 409 Conflict | "데이터 무결성 오류" | 스택 트레이스 숨김 |
| `OperationalError` | 503 Service Unavailable | "DB 연결 오류" | 스택 트레이스 숨김 |
| `asyncio.TimeoutError` | 504 Gateway Timeout | "시간 초과" | 스택 트레이스 숨김 |
| `Exception` (기타) | 500 Internal Server Error | "내부 서버 오류" | `logger.error` 기록 |

### Q: 기존 HTTPException과의 관계는?

A: FastAPI 내장 `HTTPException` 핸들러는 유지하고, 그 외 예외에 대해서만
커스텀 핸들러를 추가하는 방식이다. `HTTPException`은 이미 정형화된 응답을 반환하므로
중복 처리하지 않는다.

## 3. Rate Limiting

### Q: Rate Limiting 개요는?

A: IP 기반 Sliding Window Log 알고리즘으로 구현했다. 자세한 설계 결정은
`08_Rate_Limiting_분석.md`를 참조한다.

**핵심 동작 요약**:
- 일반 API: 100req/60s, 초과 시 `429 Too Many Requests` + `Retry-After` 헤더
- 로그인 API: 5req/60s (브루트포스 방지)
- 헬스체크: 제외 (모니터링 시스템 허용)

```
요청 →  IP 추출 → 윈도우 내 카운트 조회
             ↓                    ↓
         제한 이하           제한 초과
             ↓                    ↓
         요청 통과       429 + Retry-After 반환
```

## 4. Request Timeout

### Q: 왜 Request Timeout이 필요한가?

A: 동기 FastAPI에서 느린 DB 쿼리 하나가 스레드를 무한정 점유하면
다른 요청들이 큐에서 대기한다. 스레드 풀(기본 40개)이 전부 느린 요청으로 채워지면
새 요청은 아예 처리되지 않는다. Timeout은 이 상태를 방지하는 안전망이다.

### Q: 어떻게 구현했는가?

```python
import asyncio
from fastapi import Request
from fastapi.responses import JSONResponse

REQUEST_TIMEOUT_SECONDS = 30  # 환경변수로 조정 가능

@app.middleware("http")
async def timeout_middleware(request: Request, call_next):
    try:
        return await asyncio.wait_for(
            call_next(request),
            timeout=REQUEST_TIMEOUT_SECONDS
        )
    except asyncio.TimeoutError:
        return JSONResponse(
            status_code=504,
            content={"status_code": 504, "detail": "요청 처리 시간이 초과되었습니다."}
        )
```

`asyncio.wait_for`는 지정한 시간 내에 응답이 완성되지 않으면 `TimeoutError`를 발생시키고,
전역 에러 핸들러 또는 미들웨어에서 `504 Gateway Timeout`으로 응답한다.

| 파라미터 | 권장값 | 설정 근거 |
|---------|--------|----------|
| `REQUEST_TIMEOUT_SECONDS` | 30초 | 일반적인 웹 요청 SLA. 30초 이상은 사용자가 이미 이탈함 |
| 파일 업로드 엔드포인트 | 120초 | 대용량 업로드는 별도 상향 적용 필요 |
| 헬스체크 | 5초 | 빠른 실패가 프로브에 유리 |

### Q: 동기 엔드포인트에서의 주의사항은?

A: `asyncio.wait_for`는 코루틴에서만 동작한다. 현재 동기 엔드포인트는
FastAPI의 스레드 풀(`run_in_executor`)을 통해 실행되는데,
이 경우 timeout이 정확히 적용되지 않을 수 있다.
완전한 timeout 보장을 위해서는 비동기(`async def`) 엔드포인트 전환이 필요하다.
단기적으로는 Nginx/ALB 레벨의 `proxy_read_timeout` 설정으로 보완한다.

## 5. Circuit Breaker

### Q: Circuit Breaker가 왜 필요한가?

A: DB나 외부 서비스가 장애 상태일 때, 매 요청마다 타임아웃까지 대기하면
스레드가 낭비되고 응답 지연이 누적된다. Circuit Breaker는 장애를 감지하면
즉시 요청을 거부하여 빠른 실패(fail-fast)를 실현하고, 장애 서비스가 회복할 시간을 준다.

### Q: 3-state 상태 머신은 어떻게 동작하는가?

```
        [실패 임계값 초과]
CLOSED ─────────────────→ OPEN
  ↑                          │
  │    [회복 확인 성공]         │ [대기 시간 경과]
  └──── HALF-OPEN ←──────────┘
              │
              │ [회복 확인 실패]
              └──────────────→ OPEN (재진입)
```

| 상태 | 동작 | 전환 조건 |
|------|------|----------|
| **CLOSED** (정상) | 모든 요청 통과 | 실패율 ≥ 임계값 → OPEN |
| **OPEN** (차단) | 즉시 `503` 반환 | 대기 시간 경과 → HALF-OPEN |
| **HALF-OPEN** (탐색) | 일부 요청만 통과하여 회복 확인 | 성공 → CLOSED, 실패 → OPEN |

### Q: 설정값 기준은?

```python
class CircuitBreaker:
    failure_threshold: int = 5      # 5회 연속 실패 시 OPEN
    recovery_timeout: int = 60      # 60초 후 HALF-OPEN 전환
    success_threshold: int = 2      # 연속 2회 성공 시 CLOSED 복귀
```

| 파라미터 | 값 | 설정 근거 |
|---------|-----|----------|
| `failure_threshold` | 5 | 일시적 오류(1-2회)와 지속 장애를 구별하는 최소값 |
| `recovery_timeout` | 60초 | DB 재시작/네트워크 복구에 충분한 대기 시간 |
| `success_threshold` | 2 | 단일 성공으로 CLOSED 복귀 시 불안정한 경우 대비 |

### Q: OPEN 상태에서의 응답은?

```json
HTTP/1.1 503 Service Unavailable
Retry-After: 60

{
  "status_code": 503,
  "detail": "서비스가 일시적으로 사용 불가능합니다. 잠시 후 다시 시도해주세요."
}
```

클라이언트는 `Retry-After` 헤더 값만큼 대기 후 재시도하면 된다.
이는 Rate Limiting의 `Retry-After`와 동일한 패턴을 사용하여 클라이언트 처리를 단순화한다.

## 6. Graceful Shutdown + Health Check 분리

### Q: Graceful Shutdown이 왜 중요한가?

A: 일반적인 프로세스 종료(`SIGTERM`)는 즉시 소켓을 닫는다.
처리 중이던 요청은 클라이언트 입장에서 갑자기 연결이 끊기고,
DB 트랜잭션은 커밋/롤백 없이 중단된다. 이는 데이터 불일치를 유발할 수 있다.

### Q: active_requests 카운터 방식은 어떻게 동작하는가?

```python
active_requests = 0  # 전역 카운터
is_shutting_down = False

@app.middleware("http")
async def lifecycle_middleware(request: Request, call_next):
    global active_requests

    # 셧다운 중 새 요청 거부
    if is_shutting_down and request.url.path != "/health/live":
        return JSONResponse(status_code=503,
            content={"detail": "서버가 종료 중입니다. 다른 인스턴스로 연결하세요."})

    active_requests += 1
    try:
        return await call_next(request)
    finally:
        active_requests -= 1

async def shutdown_handler():
    global is_shutting_down
    is_shutting_down = True

    # 진행 중인 요청이 완료될 때까지 대기 (최대 30초)
    deadline = time.time() + 30
    while active_requests > 0 and time.time() < deadline:
        await asyncio.sleep(0.1)
```

셧다운 시퀀스:
1. `SIGTERM` 수신 → `is_shutting_down = True`
2. 새 요청: 즉시 `503` 반환 (로드 밸런서가 다른 인스턴스로 라우팅)
3. 진행 중 요청: 완료될 때까지 최대 30초 대기
4. 모든 요청 완료 또는 타임아웃 → 프로세스 종료

### Q: Health Check를 liveness와 readiness로 분리한 이유는?

A: Kubernetes 및 클라우드 로드 밸런서는 두 종류의 헬스 프로브를 사용한다.
이를 동일한 엔드포인트로 처리하면, 셧다운 중인 인스턴스가
"살아있음(live)"이지만 "트래픽 수신 불가(not ready)" 상태를 표현할 수 없다.

| 프로브 | 엔드포인트 | 용도 | 실패 시 동작 |
|--------|-----------|------|------------|
| **Liveness** | `GET /health/live` | 프로세스가 살아있는가? | 컨테이너 재시작 |
| **Readiness** | `GET /health/ready` | 트래픽을 받을 준비가 되었는가? | 로드 밸런서에서 제외 |

```python
@router.get("/health/live")
async def liveness():
    # 프로세스가 응답 가능한 상태인지만 확인
    # 셧다운 중에도 200 반환 (재시작 방지)
    return {"status": "alive"}

@router.get("/health/ready")
async def readiness(db: Session = Depends(get_db)):
    # DB 연결 확인 + 셧다운 여부 확인
    if is_shutting_down:
        raise HTTPException(status_code=503, detail="서버가 종료 중입니다.")
    try:
        db.execute(text("SELECT 1"))
        return {"status": "ready", "database": "connected"}
    except Exception:
        raise HTTPException(status_code=503, detail="데이터베이스 연결 불가")
```

Graceful Shutdown 중에는:
- `/health/live` → `200` (컨테이너 재시작 방지)
- `/health/ready` → `503` (새 트래픽 유입 차단)

## Q: 운영 체크리스트는?

A: 배포 전 아래 항목을 순서대로 확인한다.

### 배포 전 설정 확인

```bash
# 1. 환경변수 설정 확인
echo $DATABASE_URL          # DB 연결 문자열 (비밀번호 마스킹 확인)
echo $POOL_SIZE             # 20 이상 권장
echo $REQUEST_TIMEOUT       # 30 이상 권장
echo $RATE_LIMIT_REQUESTS   # 100 (일반), 5 (로그인)
echo $CIRCUIT_BREAKER_THRESHOLD  # 5 권장

# 2. DB 커넥션 풀 설정 확인
grep -n "pool_size\|max_overflow\|pool_timeout\|pool_recycle" app/db/session.py

# 3. 에러 핸들러 등록 확인
grep -n "exception_handler" app/main.py

# 4. 미들웨어 순서 확인 (순서가 중요)
grep -n "add_middleware\|middleware" app/main.py
```

### 배포 후 모니터링 포인트

| 지표 | 정상 범위 | 알림 임계값 |
|------|----------|-----------|
| `429` 응답 비율 | < 0.1% | > 1% |
| `503` 응답 비율 | < 0.1% | > 0.5% |
| `504` 응답 비율 | < 0.01% | > 0.1% |
| DB 커넥션 사용률 | < 70% | > 90% |
| 평균 응답 시간 | < 200ms | > 1000ms |
| Circuit Breaker 상태 | CLOSED | OPEN 전환 즉시 |

### 알림 설정 항목

- [ ] `429` 응답 1분간 100건 이상 → Slack/PagerDuty 알림
- [ ] Circuit Breaker OPEN 전환 즉시 알림
- [ ] DB 커넥션 풀 90% 이상 알림
- [ ] 5xx 응답 비율 1% 초과 알림
- [ ] 평균 응답 시간 1초 초과 알림

### 롤백 기준

아래 중 하나라도 발생하면 즉시 이전 버전으로 롤백한다:
- 5xx 응답 비율이 배포 전 대비 2배 이상 증가
- DB 커넥션 풀이 지속적으로 100% 도달
- Circuit Breaker가 OPEN 상태에서 회복하지 않음

## Q: 향후 개선 방향은?

A: 현재 구현을 바탕으로 아래 세 가지 방향으로 발전시킬 수 있다.

### 1. Redis 기반 분산 Rate Limiting

현재 인메모리 Rate Limiting은 단일 인스턴스에서만 정확하다.
인스턴스가 2개 이상이 되면 각자 독립적으로 카운트하여 실제 제한의 N배가 허용된다.

```python
# 현재: 인메모리
self._store: dict[str, list[float]] = defaultdict(list)

# 개선: Redis Sorted Set (원자적 연산)
async with redis.pipeline(transaction=True) as pipe:
    pipe.zadd(key, {str(now): now})
    pipe.zremrangebyscore(key, 0, window_start)
    pipe.zcard(key)
    pipe.expire(key, window)
    results = await pipe.execute()
    count = results[2]
```

전환 시점: 인스턴스 2개 이상 운영이 확정될 때.

### 2. Prometheus 메트릭 노출

현재는 애플리케이션 로그에서만 성능 지표를 확인할 수 있다.
`prometheus-fastapi-instrumentator`를 추가하면 Grafana 대시보드와 연동하여
실시간으로 응답 시간 분포, 요청량, 오류율을 시각화할 수 있다.

```python
from prometheus_fastapi_instrumentator import Instrumentator

Instrumentator().instrument(app).expose(app)
# GET /metrics 엔드포인트 자동 생성
```

추가 커스텀 메트릭 예시:
- `rate_limit_exceeded_total` — Rate Limit 초과 횟수 (IP별)
- `circuit_breaker_state` — Circuit Breaker 현재 상태
- `db_pool_checked_out` — 현재 사용 중인 커넥션 수

### 3. Structured Logging (구조화된 로그)

현재 로그는 평문 텍스트이므로 ELK 스택이나 Datadog에서 필터링하기 어렵다.
JSON 형식 로그로 전환하면 `request_id`, `user_id`, `response_time` 등을
필드로 쿼리할 수 있다.

```python
import structlog

logger = structlog.get_logger()

# 현재
logger.error("Unhandled exception: %s", exc)

# 개선
logger.error("unhandled_exception",
    request_id=request.headers.get("X-Request-ID"),
    path=request.url.path,
    method=request.method,
    exc_type=type(exc).__name__,
    exc_msg=str(exc))
```

### 우선순위 요약

| 개선 항목 | 전환 시점 | 예상 소요 |
|-----------|----------|----------|
| Redis Rate Limiting | 인스턴스 2개 이상 운영 시 | 1-2일 |
| Prometheus 메트릭 | 트래픽 모니터링 필요 시 | 1일 |
| Structured Logging | 로그 분석 도구 도입 시 | 0.5일 |
| 비동기 전환 | 동시 40+ 사용자 확인 시 | 2-3주 |
