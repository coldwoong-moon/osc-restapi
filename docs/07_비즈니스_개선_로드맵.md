# 비즈니스 관점 백엔드 개선 로드맵

> 이 문서는 FastAPI 전환 과정의 모든 분석 결과를 종합하여
> 비즈니스 관점에서의 개선 항목을 우선순위별로 정리한 것입니다.

---

## Q: 왜 이 종합 로드맵을 작성했는가?

A: 개별 분석(동시성, 교착, N+1, 캐시, 지연시간)은 각각의 기술적 관점에서 문제를 식별했다.
그러나 비즈니스 의사결정을 위해서는 "무엇을 먼저 해야 하는가?"와 "투자 대비 효과가 큰 것은 무엇인가?"에
답할 수 있는 종합적인 우선순위가 필요하다.

## Q: 어떤 목표를 가지고 진행했는가?

A: 6개 분석 문서의 결과를 P0(즉시)/P1(단기)/P2(중기)로 분류하고,
각 개선 항목의 예상 효과와 구현 난이도를 명시하여 프로덕션 전환 시
실행 가능한 로드맵을 제공하는 것이 목표였다.

---

## 가설 검증 결과 요약

| # | 가설 | 검증 결과 | 출처 |
|---|------|-----------|------|
| 1 | 동시 사용자가 pool_size 초과 시 응답 지연 급증 | **확인됨** — pool_size=5에서 100 동시 요청 시 큐잉으로 지연 발생 | 02_동시성_분석 |
| 2 | 역순 upsert가 교착을 유발할 수 있음 | **조건부 확인** — MariaDB InnoDB에서 행 수준 잠금 시 교착 가능. SQLite에서는 DB 레벨 잠금으로 직렬화됨 | 03_트랜잭션_교착_분석 |
| 3 | retry 전략이 일시적 잠금 경쟁을 해소함 | **확인됨** — 3회 exponential backoff retry로 OperationalError 복구 검증 | 03_트랜잭션_교착_분석 |
| 4 | 현재 CRUD 패턴에서 N+1 쿼리가 발생함 | **확인됨** — relationship lazy loading 시 N+1 발생, joinedload/selectinload로 쿼리 수 90% 감소 | 04_쿼리_최적화 |
| 5 | 캐시 적용 시 DB 부하 70% 이상 감소 가능 | **확인됨** — Pareto 접근 패턴에서 TTL 캐시 히트율 82%+ 달성 | 05_캐시_전략 |
| 6 | 콜드 스타트가 이후 요청 대비 2~5배 느림 | **확인됨** — 첫 요청 시 DB 커넥션 수립, ORM 메타데이터 로딩으로 지연 | 06_응답_지연_분석 |

## P0: 즉시 개선 (프로덕션 배포 전 필수)

### 1. 인증/세션 미들웨어 추가

**Q: 왜 이 작업이 P0인가?**
A: 현재 `/login` 엔드포인트는 사용자 정보만 반환하고 토큰/세션을 발급하지 않는다.
모든 API가 인증 없이 접근 가능하여, 프로덕션 배포 시 데이터 유출 위험이 있다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: JWT 토큰 또는 Redis 세션 기반 인증을 구현하고, `Depends()` 기반 인증 미들웨어로
보호가 필요한 엔드포인트에 인증을 강제해야 한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 보안 위험 제거, 원본 시스템 기능 대등 |
| 구현 난이도 | **중간** (JWT: 1-2일, Redis 세션: 2-3일) |
| 영향 범위 | 전체 API 엔드포인트 |

### 2. DB 커넥션 풀 설정 조정

**Q: 왜 이 작업이 P0인가?**
A: 현재 SQLAlchemy 기본값 `pool_size=5, max_overflow=10`으로 설정되어 있다.
동시 사용자 15명 초과 시 `TimeoutError`가 발생하며, 이는 서비스 중단을 의미한다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: 예상 동시 사용자 수 기반으로 `pool_size`, `max_overflow`, `pool_timeout`, `pool_recycle`을 설정해야 한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 동시 50+ 사용자 안정 처리 |
| 구현 난이도 | **낮음** (설정 변경 1시간) |
| 권장 설정 | `pool_size=20, max_overflow=30, pool_timeout=10, pool_recycle=3600` |

### 3. 전역 에러 핸들러 보강

**Q: 왜 이 작업이 P0인가?**
A: 처리되지 않은 예외(500 Internal Server Error) 시 SQLAlchemy 스택 트레이스가
응답에 노출될 수 있다. 이는 보안 취약점이자 사용자 경험 저하 요소이다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: `IntegrityError`, `TimeoutError`, `Exception` 등 미처리 예외에 대한
전역 핸들러를 추가하고, 422 Validation Error 형식도 통일해야 한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 스택 트레이스 노출 방지, 일관된 에러 응답 |
| 구현 난이도 | **낮음** (2-3시간) |

## P1: 단기 개선 (배포 후 1개월 내)

### 4. N+1 쿼리 최적화

**Q: 왜 이 작업이 P1인가?**
A: 테스트에서 N+1 발생을 확인했으며, 데이터가 증가하면 목록 조회 API의
응답 시간이 선형적으로 증가한다. 현재 데이터 규모에서는 체감되지 않지만,
프로젝트 수와 어셈블리 수가 증가하면 병목이 된다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: `joinedload`/`selectinload`를 적용하여 관계 데이터 조회 시 쿼리 수를 N+1에서 1-2로 줄인다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 목록 API 쿼리 수 90% 감소 |
| 구현 난이도 | **낮음** (CRUD get_multi에 옵션 추가) |
| 대상 | Project→ProjectModel, CarryInRequest→Items, User→Partner |

### 5. 벌크 upsert 교착 방지

**Q: 왜 이 작업이 P1인가?**
A: `upsert_part_attributes`가 입력 순서대로 레코드를 잠그므로,
두 사용자가 동시에 같은 프로젝트의 파트 속성을 수정하면 교착이 발생할 수 있다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: upsert 전에 `part_number` 기준 정렬을 적용하고, retry 래퍼를 추가한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 교착 발생 확률 제거 |
| 구현 난이도 | **낮음** (1줄 변경: `sorted(attrs_in, key=...)`) |
| 대상 코드 | `app/api/v1/parts.py:62` |

### 6. 캐시 레이어 도입

**Q: 왜 이 작업이 P1인가?**
A: 원본 시스템이 Redis 캐시를 사용했던 것은 이유가 있다.
역할 목록, 프로젝트 목록 등 변경 빈도가 낮은 데이터를 매번 DB에서 조회하는 것은
불필요한 부하이다. 캐시 적용 시 DB 부하를 70%+ 감소시킬 수 있다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: 1단계로 `functools.lru_cache` 기반 인메모리 캐시를 적용하고,
2단계로 Redis 기반 분산 캐시로 전환한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | DB 쿼리 70%+ 감소, 응답 시간 50%+ 단축 |
| 구현 난이도 | **중간** (인메모리: 1일, Redis: 2-3일) |
| 우선 캐시 대상 | roles, projects 목록, 정적 설정 데이터 |
| 권장 TTL | roles: 3600초, projects: 300초, assemblies: 60초 |

## P2: 중기 개선 (배포 후 3개월 내)

### 7. 비동기(async) 전환

**Q: 왜 이 작업이 P2인가?**
A: 현재 동기 엔드포인트는 스레드 풀(기본 40)에서 실행된다.
동시 사용자 40명 이하에서는 문제가 없지만, 그 이상에서는 큐잉이 발생한다.
비동기 전환은 I/O 바운드 처리량을 수배 향상시킬 수 있다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: `AsyncSession` + `aiomysql`로 전환하되, 측정된 병목이 있을 때만 진행한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 동시 처리량 3-5배 향상 |
| 구현 난이도 | **높음** (전체 엔드포인트 + CRUD 수정, 2-3주) |
| 전제조건 | 동시 사용자 40명 이상 트래픽 확인 |

### 8. API 버전 관리 전략

**Q: 왜 이 작업이 P2인가?**
A: 현재 `/api/v1` 단일 버전만 존재한다. 클라이언트(WinForms 등)가 배포된 상태에서
API 스키마를 변경하면 하위 호환성이 깨질 수 있다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: 버전 관리 정책(URI vs 헤더 기반)을 수립하고, 비호환 변경 시 v2 라우터를 추가할 수 있는
구조를 준비한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 클라이언트 호환성 보장 |
| 구현 난이도 | **낮음** (구조 설계 1일) |

### 9. 모니터링 및 헬스체크 강화

**Q: 왜 이 작업이 P2인가?**
A: 현재 `/health`는 단순 DB 연결만 확인한다.
프로덕션에서는 커넥션 풀 상태, 응답 시간 분포, 에러율 등을 모니터링해야 한다.

**Q: 어떤 목표를 가지고 진행해야 하는가?**
A: Prometheus metrics 엔드포인트 추가, 구조화된 로깅(JSON), 알림 연동을 구현한다.

| 항목 | 값 |
|------|-----|
| 예상 효과 | 장애 조기 감지, 성능 추이 파악 |
| 구현 난이도 | **중간** (prometheus-fastapi-instrumentator: 1일, 전체: 1주) |

## Q: 이 로드맵의 핵심 메시지는 무엇인가?

A: **"낮은 난이도, 높은 효과" 항목부터 실행하라.**

1. 커넥션 풀 설정(1시간)과 에러 핸들러(3시간)는 즉시 적용 가능하며 가장 큰 안정성 향상을 제공한다.
2. 인증 미들웨어는 보안상 필수이므로 프로덕션 배포 전 반드시 완료해야 한다.
3. N+1 최적화와 캐시 도입은 데이터 규모가 증가하기 전에 선제적으로 적용하는 것이 효과적이다.
4. 비동기 전환은 실제 트래픽 부하가 확인된 후에만 진행한다 — 불필요한 복잡성을 피한다.

## Q: 즉시 실행 가능한 코드 변경은?

A: 아래 3가지는 10분 이내에 적용 가능하다:

```python
# 1. app/db/session.py — 커넥션 풀 설정
engine = create_engine(
    settings.database_url,
    pool_pre_ping=True,
    pool_size=20,
    max_overflow=30,
    pool_timeout=10,
    pool_recycle=3600,
)

# 2. app/api/v1/parts.py:62 — 벌크 upsert 정렬
for attr in sorted(attrs_in, key=lambda x: x.part_number):

# 3. app/core/exceptions.py — 전역 500 핸들러 추가
@app.exception_handler(Exception)
async def unhandled_handler(request, exc):
    logger.error("Unhandled: %s", exc, exc_info=True)
    return JSONResponse(status_code=500, content={"status_code": 500, "detail": "Internal server error"})
```
