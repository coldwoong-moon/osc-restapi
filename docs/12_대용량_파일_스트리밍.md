# Q&A 12: 대용량 파일 스트리밍 처리

> 2026-03-11 | Spring Boot → FastAPI 전환 시 파일 처리 개선

---

## Q1. 기존 Spring Boot의 파일 업로드 방식에 무슨 문제가 있었나?

**A:** `MultipartFile.getBytes()`로 파일 전체를 `byte[]`에 적재하는 방식이었다.

```java
// 기존 방식 — FileService.java
byte[] data = multipartFile.getBytes();  // 파일 전체를 메모리에 로드
fos.write(data);                          // 한 번에 디스크에 기록
```

설정상 10GB까지 허용했지만(`DataSize.ofGigabytes(10)`), 실제로는 수백 MB만 넘어도
JVM 힙 메모리 부족(OOM)이 발생할 위험이 있었다. IFC 파일은 건축 BIM 모델로
수백 MB ~ 수 GB가 일반적이라 이 방식은 구조적 한계가 있었다.

## Q2. FastAPI에서는 어떻게 개선했나?

**A:** 청크(chunk) 단위 비동기 스트리밍으로 전환했다.

### 업로드: `aiofiles` + 청크 단위 읽기/쓰기

```python
async with aiofiles.open(file_path, "wb") as f:
    while True:
        chunk = await upload_file.read(chunk_size)  # 1MB씩 읽기
        if not chunk:
            break
        await f.write(chunk)
        total_bytes += len(chunk)
```

- 파일 크기와 무관하게 메모리 사용량이 일정 (`chunk_size`만큼만 점유)
- 10GB 파일도 1MB 메모리로 처리 가능
- `aiofiles`로 디스크 I/O도 비동기 처리 → 이벤트 루프 블로킹 방지

### 다운로드: `StreamingResponse` + async generator

```python
async def stream_file(project_id, saved_filename) -> AsyncGenerator[bytes, None]:
    async with aiofiles.open(file_path, "rb") as f:
        while True:
            chunk = await f.read(chunk_size)
            if not chunk:
                break
            yield chunk

return StreamingResponse(
    content=stream_file(project_id, saved_filename),
    media_type="application/octet-stream",
    headers={"Content-Length": str(file_size)},
)
```

- 파일 전체를 `Resource`로 메모리에 올리던 Spring Boot 방식과 달리,
  generator가 청크 단위로 `yield`하므로 메모리 사용 최소화
- `Content-Length` 헤더를 포함해 클라이언트가 진행률 계산 가능

## Q3. 왜 `aiofiles`를 사용했나? 일반 `open()`으로는 안 되나?

**A:** FastAPI는 비동기(async) 프레임워크다. 일반 `open()`은 동기 I/O라서
파일을 읽고 쓰는 동안 이벤트 루프가 블로킹된다. 이는 동시 요청 처리 성능을
크게 저하시킨다.

`aiofiles`는 내부적으로 스레드 풀에서 파일 I/O를 수행하고 결과를 비동기로
반환하므로, 이벤트 루프가 다른 요청을 처리할 수 있다.

| 방식 | 이벤트 루프 | 동시성 |
|------|------------|--------|
| `open()` (동기) | 블로킹 | 파일 I/O 중 다른 요청 대기 |
| `aiofiles` (비동기) | 논블로킹 | 파일 I/O 중 다른 요청 처리 가능 |

## Q4. 청크 크기는 어떻게 결정했나?

**A:** 기본값 1MB (`1024 * 1024 bytes`)를 선택했다.

| 청크 크기 | 장점 | 단점 |
|-----------|------|------|
| 작음 (4KB) | 메모리 최소 사용 | 시스템 콜 횟수 증가, I/O 오버헤드 |
| 중간 (1MB) | 균형 잡힌 성능/메모리 | - |
| 큼 (64MB) | I/O 오버헤드 최소 | 메모리 사용 증가 |

1MB는 대부분의 네트워크/디스크 환경에서 최적의 균형점이다.
설정(`FILE_CHUNK_SIZE`)으로 변경 가능하게 구현하여 환경에 따라 조정할 수 있다.

## Q5. 파일 크기 제한은 어떻게 구현했나?

**A:** 청크를 쓰면서 누적 크기를 체크하고, 초과 시 즉시 중단 + 파일 삭제한다.

```python
if total_bytes > settings.FILE_MAX_SIZE:
    await f.close()
    file_path.unlink(missing_ok=True)  # 불완전 파일 삭제
    raise BadRequestException(f"File exceeds maximum size")
```

Spring Boot의 `MultipartConfig`는 설정만 있고 실제 메모리 보호가 안 되었지만,
이 방식은 청크 단위로 체크하므로 초과 즉시 중단되어 리소스 낭비가 없다.

## Q6. 보안 측면에서 어떤 검증을 하나?

**A:**

1. **파일 확장자 화이트리스트**: `.ifc`, `.ifczip`, `.zip`, `.rpms`만 허용
2. **파일 크기 제한**: 청크 단위 체크로 10GB 초과 시 즉시 거부
3. **저장 파일명 랜덤화**: `secrets.token_hex(5)`로 예측 불가능한 파일명 생성
4. **JWT 인증 필수**: 모든 엔드포인트에 `Depends(get_current_user)` 적용

## Q7. Spring Boot 방식과 비교하면?

| 항목 | Spring Boot (기존) | FastAPI (개선) |
|------|-------------------|---------------|
| 업로드 메모리 | 파일 전체 (`getBytes()`) | 청크 크기만큼 (1MB) |
| 다운로드 메모리 | `Resource` 전체 로딩 | `StreamingResponse` generator |
| I/O 모델 | 동기 (블로킹) | 비동기 (`aiofiles`) |
| 크기 제한 | `MultipartConfig` 설정만 | 청크 단위 실시간 체크 |
| 파일명 보안 | 예측 가능 패턴 | `secrets.token_hex` 랜덤 |
| 진행 추적 | 없음 | `chunks_written` 반환 |
| 테스트 | 없음 | 15개 테스트 케이스 |

## Q8. 테스트는 어떻게 구성했나?

**A:** 5개 카테고리, 15개 테스트:

- **청크 업로드 (5)**: DB 레코드 생성, 청크 수 검증, 확장자 거부, 크기 제한, ifczip 허용
- **스트리밍 다운로드 (4)**: 콘텐츠 검증, Content-Length, 404, 업로드↔다운로드 무결성
- **CRUD (4)**: 빈 목록, 업로드 후 목록, 삭제(파일+DB), 없는 모델 삭제 404
- **메모리 효율 (2)**: 청크 크기별 분할 수, X-Chunk-Size 헤더

특히 **업로드→다운로드 무결성 테스트**는 원본 바이트와 다운로드 바이트가
정확히 일치하는지 검증한다.
