# JWT 인증 설계

---

## Q: 왜 JWT 인증을 도입했는가?

현재 `/login`은 사용자 정보만 반환하고 토큰이나 세션을 발급하지 않는다.
모든 API가 인증 없이 접근 가능한 상태로, 프로덕션 배포 시 데이터 유출 위험이 직접적으로 존재한다.

Spring Security 기반 원본 시스템은 인증/인가를 핵심 요구사항으로 취급하고 있으며,
기능 대등(feature parity)을 위해 FastAPI 전환 이후에도 동일 수준의 인증이 필수다.

---

## Q: 어떤 목표를 가지고 진행했는가?

1. **Stateless 인증** — 서버가 세션 상태를 유지하지 않아도 모든 요청을 인증할 수 있도록 한다.
2. **토큰 회전(rotation)** — Refresh Token을 매 갱신 시 교체하여 탈취된 토큰을 감지한다.
3. **역할 기반 접근 제어(RBAC)** — `ROLE_ADMIN`, `ROLE_USER` 등 역할에 따라 엔드포인트 접근을 제한한다.
4. **회귀 방지** — 토큰 만료, 변조, 역할 미달, 로그아웃 후 재사용 등 핵심 보안 시나리오를 테스트로 고정한다.

---

## Q: 왜 JWT를 선택했는가? (vs 서버 세션)

| 비교 항목 | JWT | 서버 세션 |
|----------|-----|----------|
| 상태 관리 | Stateless (서버 무상태) | Stateful (서버에 세션 저장) |
| 수평 확장 | 용이 (공유 세션 저장소 불필요) | Redis 등 공유 스토리지 필요 |
| 인프라 추가 | 불필요 | Redis 또는 DB 세션 테이블 필요 |
| 클라이언트 적용 | 헤더에 토큰 추가만으로 동작 | 쿠키 또는 세션 ID 관리 필요 |
| 즉시 무효화 | 불가 (만료 전까지 유효) | 가능 (서버에서 세션 삭제) |

**선택 이유**: RPMS는 WinForms 클라이언트에서 HTTP 요청 시 헤더에 토큰만 추가하면 동작하므로
JWT가 가장 적합하다. 즉시 무효화 불가 단점은 Refresh Token을 DB에 저장하여 보완했다.

---

## Q: 왜 HS256 알고리즘을 선택했는가?

단일 서버 환경에서 비대칭 키(RS256)는 불필요하다.
RS256은 공개키/개인키 쌍을 관리해야 하며, 여러 서비스가 토큰을 검증해야 하는
마이크로서비스 환경에서 가치가 있다.

**성능 비교**: HS256은 RS256 대비 서명·검증 속도가 약 10배 빠르다.
단일 서버에서 대칭키를 공유할 필요가 없으므로 HS256이 더 실용적이다.

마이크로서비스 환경으로 전환 시 RS256으로 교체할 수 있도록
`JWT_ALGORITHM` 환경변수로 알고리즘을 외부화했다.

```python
# app/core/config.py
JWT_SECRET_KEY: str = os.getenv("JWT_SECRET_KEY", "")
JWT_ALGORITHM: str = os.getenv("JWT_ALGORITHM", "HS256")
```

## Q: Refresh Token을 왜 DB에 저장하는가?

JWT의 가장 큰 단점은 **즉시 무효화 불가**다.
Access Token이 탈취되어도 만료 전까지 서버는 차단할 수 없다.
Refresh Token을 DB에 저장함으로써 세 가지 문제를 해결한다.

1. **즉시 무효화**: 로그아웃 시 DB에서 토큰을 삭제하면 갱신 요청이 차단된다.
2. **탈취 감지**: 토큰 회전(rotation) — 갱신 시 기존 토큰을 폐기하고 새 토큰을 발급한다.
   이전 토큰으로 갱신 시도가 오면 탈취로 간주하고 전체 세션을 무효화한다.
3. **서버 재시작 내구성**: 인메모리 세션과 달리 서버 재시작 후에도 세션이 유지된다.

```python
# 토큰 회전 플로우
def refresh_access_token(refresh_token: str, db: Session):
    token_record = db.query(RefreshToken).filter_by(token=refresh_token).first()
    if not token_record:
        raise HTTPException(status_code=401, detail="유효하지 않은 토큰")
    db.delete(token_record)          # 기존 토큰 폐기
    new_refresh = create_refresh_token(user_id)
    db.add(RefreshToken(token=new_refresh, user_id=user_id))  # 새 토큰 저장
    db.commit()
    return new_access_token, new_refresh
```

## Q: 토큰 만료 시간 8시간/30일의 근거는?

| 토큰 | 만료 시간 | 근거 |
|------|----------|------|
| Access Token | 8시간 | 근무 시간(8시간) 동안 재로그인 없이 사용 가능 |
| Refresh Token | 30일 | 월 1회 재로그인으로 사용자 편의성 확보 |

RPMS는 사내 시스템으로 외부 노출 위험이 낮다.
보안 민감도가 높은 시스템이라면 Access Token 15분, Refresh Token 7일로 조정할 수 있으며,
`ACCESS_TOKEN_EXPIRE_MINUTES`, `REFRESH_TOKEN_EXPIRE_DAYS` 환경변수로 외부화되어 있다.

## Q: 권한 체크를 어떻게 구현했는가?

FastAPI의 `Depends()` 패턴을 활용하여 관심사를 분리했다.

```python
# app/api/deps.py

def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: Session = Depends(get_db)
) -> User:
    """JWT 검증 + payload 추출 → User 객체 반환"""
    payload = verify_access_token(token)
    user = db.query(User).filter_by(id=payload["sub"]).first()
    if not user:
        raise HTTPException(status_code=401, detail="사용자를 찾을 수 없음")
    return user


def require_role(role: str):
    """역할 확인 의존성 팩토리"""
    def role_checker(current_user: User = Depends(get_current_user)) -> User:
        if role not in current_user.roles:
            raise HTTPException(status_code=403, detail="권한 없음")
        return current_user
    return role_checker
```

```python
# 엔드포인트 적용 예시
@router.delete("/{user_id}")
def delete_user(
    user_id: int,
    current_user: User = Depends(require_role("ROLE_ADMIN"))  # 파라미터 추가만으로 인가 적용
):
    ...
```

엔드포인트 파라미터에 의존성을 추가하는 것만으로 인증·인가가 적용된다.
라우터 로직과 인증 로직이 완전히 분리되어 유지보수가 용이하다.

## Q: 테스트 결과는?

| 테스트 그룹 | 테스트 수 | 커버 시나리오 |
|------------|---------|-------------|
| JWT 토큰 생성/검증 | 6개 | 정상 생성, 만료 토큰 거부, 변조 토큰 거부, 알고리즘 불일치 |
| 인증 플로우 | 5개 | 로그인 성공, 잘못된 비밀번호, 토큰 갱신, 로그아웃, 중복 로그아웃 |
| 의존성(Depends) | 6개 | 인증 없음, 만료 토큰, 역할 미달, 관리자 접근, 로그아웃 후 재사용 |
| **합계** | **17개** | 핵심 보안 시나리오 전체 커버 |

로그아웃 후 Refresh Token 재사용 시도, 이전 토큰으로 갱신 시도(탈취 감지),
역할 없는 사용자의 관리자 엔드포인트 접근 등 보안 관점의 경계 조건을 모두 포함한다.
